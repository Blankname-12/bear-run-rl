# 04 — Gym 仿真环境

## 1. 概述

在真机训练阶段，我们使用 Android 模拟器 + minicap 截屏 + ADB 触控的方式与真实游戏交互，单步耗时为 400ms~2000ms，每秒仅能收集约 2 帧训练样本。为了突破这一瓶颈，我们构建了纯 Python 的 Gymnasium 仿真环境 (`xiongda_env.py`)，将真机交互替换为等价的游戏逻辑模拟，使训练速度提升了三个数量级。

本章详细记录该 Gym 环境的设计思路、游戏机制推导、观测/动作空间定义、碰撞检测逻辑、奖励函数设计以及速度优势分析。

---

## 2. 设计来源

仿真环境的游戏机制来源于三类信息的交叉验证：

### 2.1 UE5 C++ 源码分析

通过逆向工程工具对游戏 APK 进行 IL2CPP 反编译，提取了以下核心 C++ 类的关键参数：

| 类名 | 提取信息 |
|------|----------|
| `ARunner` | 角色三跑道移动逻辑、速度参数、跳跃/滑铲时长 |
| `AFLOORSegment` | 地板段长度 (1000 单位)、生成行数 (3 行) |
| `AObstacleBase` | 障碍物基类及其派生类型 |
| `AWordGenerate` | 障碍物与金币的生成概率逻辑 |

### 2.2 IL2CPP 元数据字符串

从 `global-metadata.dat` 中提取的函数名和变量名字符串，补充了 UE5 源码中未覆盖的高层逻辑：

- `PlayerControl` —— 角色控制逻辑
- `ScoreRaceManager` —— 无尽跑酷分数管理
- `openMechanism1` ~ `openMechanism4` —— 四种关卡机制的触发入口
- 若干与碰撞检测、死亡判定相关的字符串

### 2.3 实际游戏观察

跳跃与滑铲的具体行为（动作动画时长、无敌判定窗口、与障碍物的碰撞结果）在 UE5 C++ 源码中不可直接读得，因此通过反复实际游戏测试和逐帧回放来推断和校准：

- 起跳到落地总时长约 0.5s
- 滑铲从开始到起身约 0.5s
- 切换跑道耗时约 0.15s（与 UE5 源码中的插值参数一致）
- 低障碍物（如栏杆）可通过跳跃安全通过
- 高障碍物（如横梁）可通过滑铲安全通过
- 地面障碍物（如地刺）需跳跃或切换跑道

## 3. 环境规范 (Gymnasium)

仿真环境严格遵循 [Gymnasium](https://gymnasium.farama.org/) API 规范。

### 3.1 观测空间

```python
gym.spaces.Dict({
    "obs":       gym.spaces.Box(low=0, high=1, shape=(4, 84, 84), dtype=np.float32),
    "lane":      gym.spaces.Discrete(3),          # 0=左, 1=中, 2=右
    "obstacles": gym.spaces.Box(low=0, high=1, shape=(10, 6), dtype=np.float32),
    "speed":     gym.spaces.Box(low=0, high=1, shape=(1,), dtype=np.float32),
})
```

| 字段 | 类型 | 维度 | 含义 |
|------|------|------|------|
| `obs` | Box | (4, 84, 84) | 4 帧堆叠的俯视灰度图，模拟运动信息 |
| `lane` | Discrete | 3 | 当前所在跑道 (0=左, 1=中, 2=右) |
| `obstacles` | Box | (10, 6) | 前方 10 个网格 (行) 各 6 个特征的障碍物信息 |
| `speed` | Box | (1,) | 归一化速度 (当前速度 / 最大速度 2000) |

**obs 表示**：采用 84x84 的俯视灰度图，模拟从角色正上方观察跑道的二维视角。画面中显示 3 条跑道、障碍物（不同灰度表示不同类型）、金币（白色圆点）。4 帧堆叠使网络能够感知运动趋势。

**obstacles 表示**：每一行 (共 10 行走廊) 有 6 个浮点数特征：

| 索引 | 特征 | 含义 |
|------|------|------|
| 0 | `has_obstacle` | 此行是否有障碍物 |
| 1-3 | `type_onehot` | 障碍物类型独热编码 (0/1/2) |
| 4 | `lane` | 障碍物所在跑道 (0/1/2) |
| 5 | `distance_norm` | 障碍物距角色的归一化距离 |

### 3.2 动作空间

```python
gym.spaces.Discrete(5)
```

| 动作编号 | 名称 | 含义 | 持续时间 |
|----------|------|------|----------|
| 0 | Jump | 跳跃，可越过低障碍物 (type 0) 和地面障碍物 (type 2) | 0.5s |
| 1 | Slide | 滑铲，可穿过高障碍物 (type 1) | 0.5s |
| 2 | Left | 向左切换跑道 | 0.15s |
| 3 | Right | 向右切换跑道 | 0.15s |
| 4 | No-op | 不执行任何动作 | — |

### 3.3 帧率

仿真环境以 30fps 速率推进，与真实游戏保持一致。每一帧执行一步仿真计算，返回一次观测。跳跃/滑铲等动作会锁定状态，在持续时间内忽略后续冲突动作。

## 4. 游戏机制 (对齐 UE5 源码参数)

### 4.1 跑道系统

```
┌─────────────┬─────────────┬─────────────┐
│  Lane 0     │  Lane 1     │  Lane 2     │
│  (左跑道)    │  (中跑道)    │  (右跑道)    │
│              │   Player    │             │
└─────────────┴─────────────┴─────────────┘
        270           270          270        (单位)
```

| 参数 | 值 | 来源 |
|------|-----|------|
| 跑道数 | 3 | UE5 `ARunner` |
| 跑道宽度 | 270 单位 | UE5 `ARunner` |
| 角色初始跑道 | 1 (中央) | 游戏默认 |

### 4.2 速度曲线

```python
base_speed      = 600         # 初始速度 (单位/秒)
max_speed       = 2000        # 最大速度 (单位/秒)
speed_increment = 2.0         # 每秒速度增量 (单位/秒²)
```

速度线性增长，每秒钟增加 2.0 单位/秒。从初始速度到最大速度需要 (2000-600)/2 = 700 秒（约 11.7 分钟）真实游戏时间。在仿真中此过程被压缩。

### 4.3 地板段与障碍物生成

| 参数 | 值 | 来源 |
|------|-----|------|
| 地板段长度 | 1000 单位 | UE5 `AFLOORSegment` |
| 生成行数 | 3 行/段 | UE5 `AWordGenerate` |
| 障碍物生成概率 | 20% | UE5 `AWordGenerate` |
| 每行最大障碍物数 | 2 | UE5 `AWordGenerate` |
| 金币生成概率 | 40% | UE5 `AWordGenerate` |
| 安全区段数 | 前 2 段 | 游戏设计惯例 |

**障碍物类型**：

| 类型编号 | 名称 | 外观 | 安全通过方式 |
|----------|------|------|-------------|
| 0 | 低障碍物 (Low Barrier) | 栏杆、低矮障碍 | 跳跃 |
| 1 | 高障碍物 (High Barrier) | 横梁、高位障碍 | 滑铲 |
| 2 | 地面障碍物 (Ground) | 地刺、地面陷阱 | 跳跃或切换跑道 |

### 4.4 动作持续时间

| 动作 | 持续时间 | 期间行为 |
|------|----------|----------|
| 跳跃 | 0.5s (15 帧) | 无法再次跳跃/滑铲；可切换跑道（跳跃中横向移动） |
| 滑铲 | 0.5s (15 帧) | 无法再次跳跃/滑铲；不可切换跑道 |
| 跑道切换 | 0.15s (~5 帧) | 可在跳跃中执行；滑铲中不可执行 |

## 5. 碰撞检测逻辑

碰撞检测以角色当前跑道位置为中心，检测前方 ±100 单位范围内的障碍物。

```python
# 伪代码
collision_range = 100  # 检测范围（单位）

for obstacle in active_obstacles:
    if obstacle.lane != player.lane:
        continue  # 不同跑道不碰撞
    if abs(obstacle.position - player.position) > collision_range:
        continue  # 超出检测范围
    # 在同一跑道且在范围内 → 检查动作是否安全
    check_collision(player, obstacle)
```

### 5.1 判定规则

| 障碍物类型 | 角色状态 | 结果 |
|-----------|---------|------|
| Type 0 (低障碍物) | 正在跳跃 | 安全通过 (DODGE) |
| Type 0 (低障碍物) | 其他状态 | 碰撞死亡 |
| Type 1 (高障碍物) | 正在滑铲 | 安全通过 (DODGE) |
| Type 1 (高障碍物) | 其他状态 | 碰撞死亡 |
| Type 2 (地面障碍物) | 正在跳跃 **或** 正在切换跑道 | 安全通过 (DODGE) |
| Type 2 (地面障碍物) | 其他状态 | 碰撞死亡 |

### 5.2 动作-障碍物兼容矩阵

|  | Type 0 (低) | Type 1 (高) | Type 2 (地面) |
|--|:--:|:--:|:--:|
| Jump | O 安全 | X 死亡 | O 安全 |
| Slide | X 死亡 | O 安全 | X 死亡 |
| Switch Lane | X 死亡 | X 死亡 | O 安全 |
| No-op | X 死亡 | X 死亡 | X 死亡 |

## 6. 奖励函数

```python
ALIVE_REWARD       = +0.1    # 每存活一步
CORRECT_DODGE      = +1.0    # 正确闪避障碍物
FAIL_DODGE         = -1.0    # 障碍物接近时不闪避 (noop)
COIN_REWARD        = +0.5    # 收集金币
DEATH_PENALTY      = -10.0   # 死亡
SCORE_FROM_COIN    = +10     # 每枚金币增加分数 (游戏内分数，非 RL 奖励)
```

### 6.1 奖励设计考量

**存活奖励 (+0.1/步)**：提供持续的正面信号，鼓励智能体维持存活状态。

**正确闪避奖励 (+1.0)**：显式奖励智能体在障碍物前执行了正确的闪避动作。这是核心学习信号。

**闪避失败惩罚 (-1.0)**：当障碍物接近而智能体选择 no-op 时施加惩罚，防止智能体"躺平"策略。

**金币收集奖励 (+0.5)**：鼓励智能体在安全的前提下收集金币，增加探索性。

**死亡惩罚 (-10.0)**：强惩罚信号，确保智能体将生存作为第一优先级。惩罚幅度大于多次存活奖励与闪避奖励之和，使单次致命失误在回报中具有显著影响。

**游戏分数**：每收集一枚金币增加 10 分游戏内分数（反映原始游戏计分规则），但此分数不直接作为 RL 奖励信号。

## 7. 速度优势

### 7.1 单步耗时对比

| 指标 | 真机 (Emulator) | Gym 仿真 |
|------|:------:|:------:|
| 单步耗时 | 400ms ~ 2000ms | < 0.1ms |
| 并行环境数 | 1 (无法并行) | 8+ |
| 每秒采集步数 | ~2 | ~10,000+ |
| 采集 1M 步 | ~6 天 | ~2 分钟 |
| 采集 5M 步 | ~30 天 | ~10 分钟 |

### 7.2 加速因子分析

真机训练的速度瓶颈主要来自：

1. **minicap 截屏延迟** (~100-200ms)：通过 ADB 获取模拟器屏幕截图
2. **图片传输延迟** (~50-100ms)：截图数据通过 ADB socket 传输
3. **推理延迟** (~10-50ms)：神经网络前向传播（取决于模型大小与硬件）
4. **ADB 触控延迟** (~50-100ms)：发送触控指令并等待执行
5. **游戏帧渲染** (~33ms/帧 @ 30fps)：模拟器内部渲染开销

而仿真环境完全在进程中完成，无任何 I/O 开销，单步计算仅为纯数学运算（矩阵运算、碰撞检测、随机生成）。

## 8. 局限性

### 8.1 视觉保真度

仿真环境使用 84x84 的简化为灰度俯视图，不包含真实游戏中的以下视觉要素：

- 3D 透视效果（近大远小）
- 动态光照与阴影
- 角色/障碍物 3D 模型细节
- 粒子特效（跑动扬尘、金币闪光）
- UI 元素（分数、血条、道具栏）
- 场景装饰（树木、建筑、天空盒）

这导致从仿真环境中学到的视觉特征无法直接迁移到真实游戏图像上。

### 8.2 难度曲线

真实游戏的难度递增并非严格的线性关系，可能存在阶段性突变（如 `openMechanism1-4` 触发更高难度机制）。仿真环境仅实现了线性速度递增，无法完全复现这些阶段性变化。

### 8.3 跳跃/滑铲机制

跳跃和滑铲的具体时机判定（如预输入窗口、动作取消规则、无敌帧精确时长）是从实际游戏观察推断得出，可能在某些边缘情况下与真实游戏存在偏差。

### 8.4 Sim-to-Real 迁移

仿真环境中训练的策略，其视觉特征提取器（CNN 部分）无法直接应用于真实游戏截图。需要额外的视觉适配层（如域随机化训练 [Domain Randomization] 或视觉特征对齐）来弥合这一差距。向量特征部分（跑道、障碍物、速度）理论上可以迁移，前提是能够从真实游戏截图中可靠地提取等价信息。

### 8.5 局限性总结

| 方面 | 仿真 vs 真机 |
|------|------------|
| 视觉输入 | 简化为 84x84 灰度俯视图 |
| 难度曲线 | 线性递增 vs 可能存在的阶段性突变 |
| 动作机制 | 推断 vs 实际编译代码 |
| 策略迁移性 | 需额外 Sim-to-Real 适配 |
| 训练效率 | 提升 > 10,000x |
