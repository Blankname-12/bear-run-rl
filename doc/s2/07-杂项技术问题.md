## 1. Ctrl+C 信号处理死循环

### 问题现象

用户按 Ctrl+C 期望程序优雅退出，实际表现为：
- 每次按 Ctrl+C 打印一行 `[Train] 收到中断信号，正在安全退出...`
- 设置 `interrupted = True` 后，外层 `while not interrupted` 循环仍然继续
- 反复按 Ctrl+C 反复打印，看起来像"程序死循环，Ctrl+C 无效"
- 只能通过任务管理器杀进程

### 根因分析

```python
# === 原始有 BUG 的代码 ===
interrupted = False

def signal_handler(sig, frame):
    global interrupted
    print("\n[Train] 收到中断信号，正在安全退出...")
    interrupted = True

# 训练主循环
while not interrupted:
    obs = env.reset()
    done = False
    while not done:
        action = agent.act(obs)
        next_obs, reward, done, info = env.step(action)
        agent.learn(...)
        obs = next_obs

    # 一个 episode 结束后（done=True），即使 interrupted=True，
    # 这段代码仍然会执行：
    print_episode_stats()       # 打印统计
    agent.save_model()          # 保存模型
    env.restart_game()          # 重启游戏 (耗时 ~3.5s)

    if interrupted:
        break                   # 循环检查在这里，但已经太晚了
```

问题分析：

| 阶段 | 耗时 | 说明 |
|------|------|------|
| `print_episode_stats()` | ~0.1s | 打印本局统计 |
| `agent.save_model()` | ~0.5s | 写入 PyTorch 权重文件 |
| `env.restart_game()` | **~3.5s** | 等待 + ADB tap 序列 |
| 合计 post-episode | ~4.1s | 用户 4 秒内连按 5-6 次 Ctrl+C |

当用户在 post-episode 期间连按 Ctrl+C 时：
1. `interrupted = True` 但仍在 `restart_game()` 中（sleep + ADB tap）
2. `restart_game()` 成功后回到 while 检查 `if interrupted: break`
3. 如果这个 break 因为某些原因没有生效（如嵌套循环结构），游戏继续被 restart 了
4. 用户看到游戏被重启 → 以为中断没生效 → 再按 Ctrl+C → 循环

### 修复方案

```python
# === 修复后的代码 ===
_interrupt_count = 0

def signal_handler(sig, frame):
    global interrupted, _interrupt_count
    _interrupt_count += 1
    if _interrupt_count == 1:
        print("\n[Train] 收到中断信号，正在安全退出... (再按一次 Ctrl+C 强制退出)")
        interrupted = True
    else:
        print("\n[Train] 强制退出！")
        os._exit(1)  # 直接终止进程，不执行任何清理
```

同时在 post-episode 代码块开头立即检查：

```python
# Episode 结束后立即检查
if interrupted:
    break  # 跳过 stats 打印、模型保存、游戏重启

# 游戏重启加保护
if done and not interrupted:
    env.restart_game()
```

关键改进：
1. **双击强制退出**：第一次 Ctrl+C 优雅退出，第二次 `os._exit(1)` 直接杀进程，绝不兜圈
2. **post-episode 立即 break**：不打印统计、不保存模型、不重启游戏
3. **重启加 guard**：只在 `not interrupted` 时才重启游戏

## 2. ADB 中文路径问题

### 问题现象

```bash
$ adb shell ls /sdcard/xd.apk
E:/git/sdcard/xd.apk: No such file or directory
```

ADB 在执行 shell 命令前自动在路径前追加了 `E:/git/` 前缀。

### 根因分析

该系统上 ADB 通过 MSYS2/Cygwin 路径转换层调用，shell 解释器将 Linux 风格路径 `/sdcard/...` 错误地转换为 Windows 路径 `E:/git/sdcard/...`（因为当前工作目录恰好在 `E:/git` 下）。这是 MSYS2 的 automatic POSIX path conversion 导致的问题，而非 ADB 自身的 bug。

### 解决方案

使用 **PowerShell** 执行 ADB 命令（而非 Bash/MSYS2），PowerShell 不会对路径执行 POSIX 转换：

```powershell
PS> adb shell ls /sdcard/
```

或在 Bash 中通过环境变量临时关闭 MSYS2 路径转换：

```bash
MSYS2_ARG_CONV_EXCL="*" adb shell ls /sdcard/
```

本实验中统一采用 PowerShell 执行所有 ADB 截图和触控命令。

## 3. Windows GBK 编码问题

### 问题现象

```python
print("训练状态: 第 1 局, 得分 42")
# UnicodeEncodeError: 'gbk' codec can't encode character '局' in position 5
```

### 根因分析

Windows 中文版系统的默认终端编码是 **GBK (cp936)**，而 Python 3 默认使用 UTF-8 处理字符串。当 `print()` 尝试将 UTF-8 字符串输出到 GBK 终端时，Python 内部的 `sys.stdout` 会调用 `encode('gbk')`，遇到 GBK 不支持的 Unicode 字符时抛出 `UnicodeEncodeError`。

注意：这个报错发生在 `sys.stdout.buffer.write()` 内部，不是在 `print()` 调用处，因此 `try/except` 无法捕获。

### 解决方案

```python
import sys
import io

sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8', errors='replace')
sys.stderr = io.TextIOWrapper(sys.stderr.buffer, encoding='utf-8', errors='replace')
```

关键参数：
- `encoding='utf-8'`：强制 stdout 使用 UTF-8 编码输出
- `errors='replace'`：遇到终端无法显示的字符时替换为 `?` 而非抛出异常

如果终端仍然乱码（Windows cmd/PowerShell 默认字库不包含所有 CJK 字形的 UTF-8 渲染），可在终端设置中切换到支持 UTF-8 的字体（如 Consolas、Cascadia Code），或设置 `chcp 65001` 切换代码页。

## 4. Conda 环境与中文路径

### 环境信息

```bash
Conda 环境: D:\anaconda\anaconda3\envs\RL\python.exe
主要依赖: torch, numpy, opencv-python, pillow
```

### 问题现象

```bash
$ conda run -n RL python train.py
# conda run 内部在执行过程中遇到中文路径时报 UnicodeEncodeError
```

### 根因分析

`conda run` 会修改进程的环境变量和工作目录上下文。当工作目录或脚本路径包含中文字符（本项目路径 `D:\一些word\实验报告\大三下\强化学习\...` 就在广泛的 UTF-8 路径下），`conda run` 的内部 subprocess 调用在 Windows 上使用非 UTF-8 的默认编码传递路径，导致路径字符串被截断或乱码。

### 解决方案

不使用 `conda run -n RL`，直接调用环境内的 Python 可执行文件：

```bash
D:\anaconda\anaconda3\envs\RL\python.exe train.py
```

或使用绝对路径：

```python
# train.py 内部指定专用 Python
#!/usr/bin/env python
# 运行时: & "D:\anaconda\anaconda3\envs\RL\python.exe" train.py
```

在 VS Code 中，将 Python 解释器路径设置为 `D:\anaconda\anaconda3\envs\RL\python.exe` 即可避免所有 conda run 相关问题。

## 5. pygame 与 Python 3.12 不兼容

### 问题现象

```python
import pygame
# AttributeError: module 'pkgutil' has no attribute 'ImpImporter'
```

### 根因分析

Python 3.12 从 `pkgutil` 模块中移除了 `ImpImporter` 类（该类在 Python 3.3 导入系统重构后已被标记废弃，3.12 正式删除）。旧版 pygame 的初始化代码仍然引用 `pkgutil.ImpImporter`，导致 `import pygame` 立即崩溃。

### 解决方案

放弃 pygame，改用 **OpenCV (cv2)** 进行可视化渲染。

本项目中的 `visualize.py` 原本使用 pygame 做窗口创建、图形绘制和键盘事件处理。替换方案：

| 功能 | pygame 原实现 | OpenCV 替换 |
|------|--------------|-------------|
| 窗口创建 | `pygame.display.set_mode()` | `cv2.namedWindow()` |
| 绘制图形 | `pygame.draw.rect/circle()` | `cv2.rectangle/circle()` |
| 渲染文本 | `pygame.font.render()` | `cv2.putText()` |
| 键盘输入 | `pygame.event.get()` | `cv2.waitKey()` |
| 帧率控制 | `pygame.time.Clock()` | `time.sleep()` / `cv2.waitKey(n)` |

替换后代码量相近，性能无明显差异。

## 6. OpenCV 可视化 (visualize.py)

### 功能概述

`visualize.py` 是本项目的独立可视化模块，提供以下两种模式：

- **Manual 模式**：W/A/S/D 或方向键控制角色（左移/右移/跳跃/滑铲），体验真实游戏手感
- **AI 模式**：加载训练好的 PyTorch 模型，agent 自动决策动作，实时渲染

键盘快捷键：
- `Q` — 退出
- `R` — 重启游戏
- `Space` — 暂停/继续
- `M` — 切换 Manual/AI 模式

### 渲染技术细节

**3D 透视投影**

可视化采用伪 3D 俯视视角，将世界坐标投影到 2D 屏幕：

```
world_x (forward distance, 前方距离) → screen_y (越远越靠上)
lane_offset (横向偏移)              → screen_x
height_scale (深度缩放)              → 近大远小
```

投影函数：
```
proj_x(lane, distance) = screen_center_x + lane_offset * height_scale(distance)
proj_y(distance)       = horizon_y + (max_distance - distance) / max_distance * ground_height
```

**绘制元素**

| 元素 | 颜色 (BGR) | 形状 | 说明 |
|------|-----------|------|------|
| 天空 | (255, 180, 100) | 矩形渐变 | 浅蓝色天空 |
| 地面 | (80, 180, 80) | 矩形 | 绿色草地 |
| 道路 | (80, 80, 100) | 梯形 | 透视梯形，近宽远窄 |
| 车道线 | (255, 255, 255) | 虚线 | 沿透视方向等距分布 |
| 玩家 | (255, 200, 50) | 矩形 | 底部居中显示 |
| 障碍物 1 | (0, 0, 255) | 红色矩形 | 正面障碍，需跳跃 |
| 障碍物 2 | (255, 0, 0) | 蓝色矩形 | 高位障碍，需滑铲 |
| 障碍物 3 | (255, 0, 255) | 品红矩形 | 低位障碍，需滑铲 |
| 金币 | (255, 220, 50) | 黄色圆形 | 得分道具 |

**角色动画**

- 跳跃动画：玩家 Y 坐标向上偏移 40px，持续 6 帧后回落
- 滑铲动画：玩家 Y 坐标向下偏移 10px，矩形高度压缩为 50%

**HUD (Heads-Up Display)**

```
+------------------------------------------+
|  Distance: 542m  |  Coins: 13  |  Score: 1042 |
|  Speed: 1.2x     |  Lane: L    |  Status: RUN  |

```

**死亡画面**

当角色碰撞障碍物触发 `done=True` 时，画面叠加半透明红色遮罩，中央显示白色大字 "WASTED!"，持续 2 秒后自动重启或返回菜单。

## 7. 数据采集最终架构（极简版）

经过多轮迭代和简化后，数据采集的核心逻辑缩减为以下极简结构：

```python
import subprocess, io, time
import numpy as np
from PIL import Image
from pynput import keyboard

recorded_data = []  # 每项: (state_image, raw_jpg_bytes, action)

def screencap():
    """单行截屏: ADB exec-out screencap -p"""
    r = subprocess.run(
        ["adb", "-s", SERVER, "exec-out", "screencap", "-p"],
        capture_output=True, timeout=10
    )
    img = Image.open(io.BytesIO(r.stdout)).convert("RGB")
    return np.array(img), r.stdout  # 返回 numpy 和原始 JPG

# 主循环
def on_press(key):
    """pynput 键盘回调: 记录动作"""
    action = key_to_action(key)  # W→jump, S→slide, A→left, D→right, 其他→none
    state_img, raw_jpg = screencap()          # 1. 先截图 (反应前状态)
    recorded_data.append((state_img, raw_jpg, action))  # 2. 保存
    send_action_to_device(action)             # 3. 发送触控

listener = keyboard.Listener(on_press=on_press)
listener.start()
listener.join()
```

**架构要点**：

- **无 GameEnv 类**：不再封装 `reset()` / `step()` / `observe()` 等 Gym 接口
- **无 minicap**：不启动 minicap 进程、不设置 ADB forward
- **无多线程**：不启动截屏线程、不启动心跳检测线程、不用队列
- **无死亡检测**：不检测红心区域是否为空（该模块在训练阶段才需要）
- **截屏 → 记录 → 发送动作** 严格按此顺序，保证 `(state, action)` 的因果一致性

**为何如此极简？**

数据采集阶段的核心目标是**快速稳定地收集大量 (state, action) 轨迹**，而非实时控制。保持程序极简的好处：
1. 崩溃风险低（代码越少，bug 表面积越小）
2. 调试容易（15 行主循环 vs 300+ 行 GameEnv）
3. 性能可预测（每一步的耗时 = 截屏时间 + 保存时间，无其他异步开销）

## 8. Gym 环境与真实游戏的差异

### UE5 源码分析结果

反编译 APK 中的 UE5 (Unreal Engine 5) 游戏逻辑源码得出以下发现：

**游戏源码实际实现的动作**：
- ✅ 左移 (MoveLeft)
- ✅ 右移 (MoveRight)
- ❌ 跳跃 (Jump) — 源码中未找到对应实现
- ❌ 滑铲 (Slide) — 源码中未找到对应实现

**APK 字符串表中存在的动作**：
```
Jump              ← 字符串存在
Slide             ← 字符串存在
openMechanism1    ← 字符串存在 (开机关 1)
openMechanism2    ← 字符串存在 (开机关 2)
openMechanism3    ← 字符串存在 (开机关 3)
openMechanism4    ← 字符串存在 (开机关 4)
```

### 数据分析

| 来源 | 左移 | 右移 | 跳跃 | 滑铲 | 开机 | 说明 |
|------|------|------|------|------|------|------|
| UE5 源码 | 有 | 有 | 无 | 无 | 无 | 可能是简化后的版本 |
| APK 字符串表 | 有 | 有 | 有 | 有 | 有 | 字符串资源表包含完整动作名 |
| 实际游戏 | 有 | 有 | 有 | 有 | ? | 肉眼观察确认存在跳跃和滑铲 |

可能的解释：
1. 反编译不完全，跳跃/滑铲逻辑分布在未成功反编译的 UE Blueprint 模块中
2. APK 中打包了完整的 UE5 运行时，但具体玩法逻辑通过 Blueprint (可视脚本) 实现，而非 C++ 源码——Blueprint 反编译困难，未能完全还原
3. 字符串表可能是 UE5 引擎的通用字符串（用于多款游戏），部分字符串仅作预留

### 对 RL 环境模型的影响

本项目的 `xiongda_env.py` (Gym 环境) 中实现了完整的 4 个动作：

```python
class XiongdaEnv(gym.Env):
    action_space = spaces.Discrete(4)
    # 0: LEFT, 1: RIGHT, 2: JUMP, 3: SLIDE
```

跳跃和滑铲的逻辑是基于以下推断添加的：
- 肉眼观察真实游戏确认存在这两类动作
- 动作时序和碰撞判定基于帧间分析近似获得
- 跳跃/滑铲的奖励函数为经验性设定（未经源码验证）

**这是一个已知的局限性**，需要在实验报告中明确说明：跳跃和滑铲的状态转移函数不是从真实游戏源码中精确提取的，而是基于观察和推断构建的近似模型。这可能会影响：
1. **仿真与真实游戏的策略迁移效果**：在 `xiongda_env.py` 中训练的策略，部署到真实游戏时，跳跃和滑铲的成功率可能低于预期
2. **奖励函数的准确性**：如果真实游戏的碰撞检测与模拟环境不同，reward 会有天然偏差
