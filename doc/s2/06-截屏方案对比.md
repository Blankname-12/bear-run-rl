# 06 — 截屏方案对比

> 实验环境：Windows 11 + LDPlayer Android 9 模拟器 (x86_64, SDK 28) + ADB  
> 需求：从模拟器实时截取 1080x1920 游戏画面，传递给 RL agent 做状态观测

---

## 方案 1: minicap (最优先尝试，最终不稳定)

### 原理

[minicap](https://github.com/DeviceFarmer/minicap) 是 STF (Smartphone Test Farm) 项目出品的 Android 原生截屏工具。它绕过 Android Framework 的 `screencap` 命令，直接调用 SurfaceFlinger 底层 API 获取帧缓冲，因此可以达到远高于 ADB screencap 的帧率。

minicap 通过 UNIX domain socket（抽象套接字 `@minicap`）对外暴露帧流，客户端通过 ADB forward 将本地 TCP 端口映射到该 socket，然后自行解析二进制协议帧即可逐帧读出 JPEG 压缩图像。

### 二进制与部署

```
文件:
  bin/x86_64/minicap          → /data/local/tmp/minicap
  shared/android-28/x86_64/minicap.so  → /system/lib64/minicap.so (需 root + remount)

启动命令:
  adb shell "nohup /data/local/tmp/minicap -P 1080x1920@1080x1920/0 -r 15 -S >/dev/null 2>&1 &"

ADB 端口转发:
  adb forward tcp:1717 localabstract:minicap
```

参数说明：
- `-P 1080x1920@1080x1920/0`：物理分辨率 1080x1920，输出分辨率也是 1080x1920，旋转 0 度
- `-r 15`：限制最大输出帧率 15fps
- `-S`：静默模式，不输出调试信息到 stdout

### 协议格式

**全局头部（24 字节，LE 编码）**

| 偏移 | 大小 | 字段 |
|------|------|------|
| 0 | 4 | 版本号 |
| 4 | 4 | banner 长度 |
| 8 | 4 | 显示宽度 (px) |
| 12 | 4 | 显示高度 (px) |
| 16 | 4 | 输出宽度 (px) |
| 20 | 4 | 输出高度 (px) |
| 24+ | ? | banner 字符串 + 后接旋转角度、flags |

**帧格式**

| 字段 | 大小 |
|------|------|
| 帧大小 (LE uint32) | 4 字节 |
| JPEG 数据 | 帧大小字节 |

每帧 = 4 字节 LE 大小 + 紧接 JPEG 二进制流。客户端读取头部后按大小读满一帧即可。

### 性能

| 指标 | 数值 |
|------|------|
| 端到端延迟 | ~50ms/帧 |
| 最高帧率 | 15fps (被 `-r 15` 限制) |
| 分辨率 | 原生 1080x1920 |
| JPEG 码率 | ~600KB/帧 (15fps 下 ~9MB/s) |

**为何限制 15fps？** 若设为 60fps，ADB channel 会被 9MB/s 的 JPEG 流占满，导致 touch 指令延迟飙升至 1 秒以上，agent 的控制响应严重滞后，实际游戏体验和 RL 训练效果都会恶化。15fps 是在画面流畅度和控制延迟之间的权衡。

### `get_frame()` 优化：帧丢弃策略

minicap 会持续生产帧并堆积在 socket 缓冲区中。当 agent 调用 `get_frame()` 时，可能已有 5-8 帧旧数据排队。若逐一解码所有旧帧再取最后一帧，每帧 JPEG 解码耗时约 10ms，累计 ~50-80ms。

优化方法：**只读不解码** —— 逐帧从 socket 读出 size + JPEG blob 但跳过 `Image.open` 解码步骤，仅保留最后一帧的 JPEG 数据进行解码：

```
1. poll socket 直到无新数据可读
2. 对每帧: 读 4 字节 size → 读 size 字节 JPEG → 丢弃 (不 decode)
3. 对最后一帧: 保留 JPEG blob → Image.open → np.array
```

此优化将 `get_frame()` 的额外延迟控制在 ~5ms 内。

### 故障模式

```
[Frame]  解析错误: 70778880x125829120 (chunk 1, size=363776)
[Frame]  解析错误: 65536x1633158144 (chunk 2, size=421760)
[Frame]  解析错误: 2166398976x1083494912 (chunk 3, size=524224)
```

minicap 在运行过程中随机崩溃，返回的全局头部分辨率字段为垃圾值（70778880 x 125829120，即 `0x0437F000 x 0x07800000`），显然是内存越界或未初始化内存被写入 header 区域。

进一步诊断：minicap 进程在崩溃后彻底消失（`adb shell ps | grep minicap` 返回空），需要完全重新部署整套流程（kill 残留 socket、重新 push 二进制、重新 forward 端口）。

### 根因分析

LDPlayer 运行的是 x86_64 架构的 Android 9 (SDK 28)，模拟器中 NDK 翻译层 (libhoudini/libnb) 存在兼容性问题。minicap 通过 Android NDK 的 `libgui.so` → `libsurfaceflinger.so` → `SurfaceComposerClient` 链路直接访问帧缓冲，该路径在模拟器环境中不稳定。表现特征：
- 有时能正常运行 5-10 分钟
- 有时启动后 3 秒内崩溃
- 崩溃后无 core dump，无 logcat 异常栈
- header 字段被随机字节覆盖（典型的内存踩踏/use-after-free）

市面部分 Android 模拟器（如 MuMu、Nox）在 x86 Android 7 下曾成功运行 minicap，但 LDPlayer Android 9 的兼容性问题在此实验中未能解决。

## 方案 2: Windows DXGI Desktop Capture (窗口模式)

### 原理

使用 Python 的 [mss](https://github.com/BoboTiG/python-mss) 库，底层调用 Windows DXGI (DirectX Graphics Infrastructure) Desktop Duplication API，直接从 GPU 后台缓冲区获取桌面/窗口的像素数据。零拷贝设计使其成为除 GPU 硬件编码外最快的截屏方案。

### 步骤

```python
import mss
import win32gui
import numpy as np

def get_game_window_rect():
    """遍历顶层窗口, 按标题关键词匹配模拟器窗口"""
    def enum_callback(hwnd, results):
        if win32gui.IsWindowVisible(hwnd):
            title = win32gui.GetWindowTextW(hwnd)
            if any(kw in title for kw in ("LDPlayer", "雷电", "熊大")):
                rect = win32gui.GetWindowRect(hwnd)
                results.append((hwnd, title, rect))

    windows = []
    win32gui.EnumWindows(enum_callback, windows)
    return windows[0][2] if windows else None

def capture():
    rect = get_game_window_rect()  # e.g. (100, 50, 458, 724)
    monitor = {"top": rect[1], "left": rect[0],
               "width": rect[2] - rect[0], "height": rect[3] - rect[1]}
    with mss.mss() as sct:
        img = sct.grab(monitor)  # BGRA numpy array
        return np.array(img)[:, :, :3]  # strip alpha
```

### 性能

| 指标 | 数值 |
|------|------|
| 端到端延迟 | ~17ms/帧 |
| 帧率上限 | ~58fps (受屏幕刷新率制约) |
| ADB 依赖 | 无（完全独立于模拟器） |
| 额外软件 | 无（只需 mss + win32gui） |

### 优势

- **最快**：17ms 延迟是所有方案中最优
- **零模拟器依赖**：不需要 push 任何二进制文件、不需要 root、不需要 remount
- **不需要 ADB**：数据路径完全不经过 ADB channel，不影响 touch 控制通道

### 故障模式

```
图像尺寸: 358x674  (期望 1080x1920)
缩放因子: X=0.331, Y=0.351
```

模拟器窗口在桌面上仅显示 358x674 像素（约为 1080x1920 的 1/3 缩放），而游戏内部实际以 1080x1920 渲染。这导致两个致命问题：

1. **分辨率损失**：图像细节、文字、图标缩小到 1/3 后，OpenCV 特征检测（如红心位置检测）完全失效
2. **坐标转换错误**：需要将检测到的物体坐标从 1080x1920 空间映射到 358x674 → 桌面绝对坐标 → ADB tap 坐标。非线性缩放因子（X 和 Y 缩放不同）意味着映射误差在 Y 方向可达 20-30 像素

### 实际表现

用户反馈截图结果为"就截了一个小角落，而且还有 20% 的电脑屏幕"。这是因为 `mss.grab(monitor)` 捕获的是桌面坐标系下的矩形区域：
- 模拟器窗口边框并非纯净的游戏画面（包含标题栏、菜单栏）
- 窗口定位受到桌面 DPI 缩放影响（Windows 125% / 150% 缩放会使坐标进一步偏移）
- 窗口必须保持可见且不被其他窗口遮挡

### 前提条件

- 模拟器窗口必须保持**可见**（不能最小化，不能被其他窗口遮挡）
- 窗口在屏幕上的像素尺寸必须与游戏渲染分辨率 1:1 映射（或已知精确比例）
- 对于 LDPlayer 默认窗口大小（~358x674，不可拉伸到 1080x1920），此方案无法满足需求

## 方案 3: ADB exec-out screencap (最稳定，速度最慢)

### 原理

Android 自带的 `screencap` 命令读取 `/dev/graphics/fb0`（帧缓冲设备）并输出为 PNG 或原始 RGB 数据。`exec-out` 模式通过 ADB 的 `shell v2` 协议直接返回 stdout，绕过 PTY（伪终端），避免终端控制字符污染二进制数据。

```bash
adb -s <device> exec-out screencap -p   # PNG via stdout, 无 PTY 包装
```

### 原始实现（过度复杂）

早期的 `_adb_screencap` 函数实现了 **3 种方法 x 3 次重试** 的冗余逻辑：

| 方法 | 描述 | 问题 |
|------|------|------|
| Method 0 | `adb exec-out screencap -p` | 最快 |
| Method 1 | `adb shell screencap -p /sdcard/tmp.png && adb pull` | 回退方案 1 |
| Method 2 | `adb exec-out screencap` (raw RGB) + 手动解析 + cv2.cvtColor 转 BGR | 回退方案 2 |

每种方法有 10 秒独立超时，最坏情况下 **3 种方法 x 3 次重试 x 10 秒 = 90 秒** 才能返回失败。实际上仅 Method 0 有效，Method 1 和 2 从未成功过。

此外，每 200 次截屏后会主动重启 adbd 以避免潜在的内存泄漏：

```python
ADBD_RESTART_INTERVAL = 200  # 每 200 帧重启 adbd
```

### 优化后方案

将截屏逻辑简化为**单行命令 + 单次重试 + 8 秒超时**：

```python
import subprocess
import io
import numpy as np
from PIL import Image

def screencap(device_serial):
    for _ in range(2):  # 最多 2 次尝试
        try:
            r = subprocess.run(
                ["adb", "-s", device_serial, "exec-out", "screencap", "-p"],
                capture_output=True, timeout=8
            )
            if r.returncode == 0 and len(r.stdout) > 1000:
                img = Image.open(io.BytesIO(r.stdout)).convert("RGB")
                return np.array(img)
        except (subprocess.TimeoutExpired, Exception):
            continue
    return None
```

### 性能对比

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 每次耗机 | 2+ 秒 | ~400ms |
| 有效帧率 | <1fps | ~2fps |
| 超时最坏情况 | 30+ 秒 | 8 秒 |
| 代码行数 | ~120 行 | ~15 行 |

### 为何可用

- **约 2fps**：虽然在游戏流畅度上很慢，但用于**离线数据采集**（人工玩、程序只录屏 + 录操作）已足够
- **完整分辨率**：无论模拟器窗口多大，始终输出原生 1080x1920
- **100% 稳定**：整个实验周期从未崩溃或返回垃圾数据
- **PNG 无损**：不引入 JPEG 压缩伪影，适合后续用于训练的视觉特征提取

## scrcpy (未重新尝试)

[scrcpy](https://github.com/Genymobile/scrcpy) 是目前最完善的 Android 屏幕镜像方案，具备以下特点：
- H.264 硬件编码视频流 → 客户端 ffmpeg 解码
- 极低延迟（35-70ms 端到端）
- 成熟稳定

在本实验的**上一轮对话 (Session 1)** 中曾尝试集成 scrcpy 作为截屏后端，遇到两个技术障碍：

1. **PyAV 解码失败**：Python 的 PyAV (ffmpeg binding) 无法正确解析 scrcpy 输出的原始 H.264 NAL 单元流，因为它缺少容器级 metadata
2. **ffmpeg pipe 挂起**：通过 subprocess 调用 `ffmpeg -i - -f rawvideo -` 管道时，ffmpeg 持续等待第一个 IDR (Instantaneous Decoder Refresh) 帧后才开始输出，而等待期间无任何数据、无法设置超时，导致管道永久阻塞

因此本轮实验**未再尝试 scrcpy**。

## 方案对比总结

| 方案 | 端到端延迟 | 有效帧率 | 输出分辨率 | 稳定性 | 优缺点 |
|------|-----------|----------|-----------|--------|--------|
| **minicap** | ~50ms | ~15fps | 原生 1080x1920 | 崩溃 (x86_64 不兼容) | 理想方案，但不可用 |
| **Window DXGI (mss)** | ~17ms | ~58fps | 窗口大小 (358x674) | 区域错误 | 最快，但缩放坐标全错 |
| **ADB exec-out (优化后)** | ~400ms | ~2fps | 原生 1080x1920 | 稳定 | 慢但可用 |
| **ADB exec-out (优化前)** | 2+ 秒 | <1fps | 原生 1080x1920 | 稳定 | 太慢，须优化 |
| **scrcpy** | ~70ms | ~30fps | 原生 1080x1920 | 未测通 | 解码管道阻塞，未采用 |

---

## 最终方案选择

**数据采集阶段选用 ADB exec-out screencap -p (优化后的极简单行方案)**。

理由：
1. **~2fps 足够用于离线数据采集**：用户手动玩游戏，程序只负责同步录制 `(state_image, user_action)` 对
2. **24/7 稳定运行**：整个采集周期无崩溃、无数据损坏
3. **原生分辨率**：1080x1920 图像保证后续所有 OpenCV 特征检测（红心区域、障碍物识别、距离判断）精度达标
4. **极简实现**：15 行代码，无第三方依赖（PIL + numpy 是项目标配），出问题容易排查

如果后续阶段需要实现 **实时 RL agent 控制**（而非离线采集），需要重新评估截屏方案。届时 scrcpy 的 H.264 解码问题需要通过 `ffmpeg` 的 `-probesize` / `-analyzeduration` 参数重新调适，或使用 scrcpy 的控制 socket 单独发送下行命令。
