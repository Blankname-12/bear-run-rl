## 2. 真机训练 — train.py (已弃用)

### 2.1 架构概述

`train.py` 是整个项目的第一代训练代码，直接在 Android 模拟器上运行，通过 minicap 截屏与 ADB 触控实现闭环交互。

```text
┌──────────┐   截屏    ┌──────────┐   图像    ┌──────────┐   动作
│ Android  │ ──────→ │ minicap  │ ──────→ │  PPO     │ ──────→ │ ADB 触控 │
│ Simulator│ ←────── │ 截图传输  │         │  Agent   │ ←────── │ 指令下发 │
└──────────┘          └──────────┘         └──────────┘          └──────────┘
```

### 2.2 网络架构

采用 ResNet-18 作为视觉骨干网络 (~11M 参数)：

- 输入：15 帧堆叠的 320x320 RGB 截图
- ResNet-18 卷积层提取空间特征
- 空间注意力模块 (Spatial Attention) 聚焦关键区域
- Actor 头：全连接层 → 5 维动作 logits
- Critic 头：全连接层 → 1 维状态价值

### 2.3 死亡检测

由于模拟器环境无法直接通过 API 获取游戏状态，死亡判定完全依赖视觉信号：

**HSV 心脏检测**：
- 游戏界面上方的血量条（心脏图标）颜色范围：H ∈ [0, 10] ∪ [170, 180], S ∈ [100, 255], V ∈ [100, 255]
- 在血量区域进行 HSV 掩码检测，统计红色像素占比
- 心脏红色像素数低于阈值 → 判定为死亡

**3 帧投票机制**：
- 连续 3 帧检测到心脏缺失 → 确认死亡（防止单帧误检）
- 此机制有效抑制了因瞬间画面闪烁、截图抖动导致的误判

**verify_death 二次确认**：
- 死亡触发后执行一次额外截图验证
- 确认画面停留在死亡界面（通过检测"再来一次"按钮的颜色模式）
- 防止因短暂画面遮挡导致的误触发

### 2.4 自动重启机制

角色死亡后，训练流程需要自动重启游戏以继续收集样本：

```text
1. 等待死亡动画结束 (~1s)
2. 点击"再来一次"按钮 (坐标比例 0.5, 0.7)
3. 等待加载界面 (~1s)
4. 点击"开始游戏"按钮 (坐标比例 0.5, 0.6)
5. 等待游戏开始 (~1.5s)
6. 重置环境状态，继续训练
```

完整重启周期约 3.5 秒。在 24 小时不停机训练中，此开销占训练总时间约 15-20%（取决于死亡频率）。

### 2.5 Ctrl+C 信号处理

初版 `train.py` 存在一个关键 bug：训练主循环使用 `while True` 且未正确处理 `SIGINT`，导致 Ctrl+C 无法正常中断训练进程，只能通过 `taskkill` 强制终止。

**修复方案**：
```python
import signal
import sys

interrupted = False

def signal_handler(sig, frame):
    global interrupted
    interrupted = True
    print("\n[INFO] Ctrl+C received. Saving checkpoint and exiting...")

signal.signal(signal.SIGINT, signal_handler)

# 训练主循环
while not interrupted:
    # ... 训练步骤 ...
```

收到中断信号后自动保存 checkpoint 并优雅退出，确保不丢失已训练的模型权重。

### 2.6 Checkpoint / Resume 机制

```python
checkpoint = {
    'model_state_dict': agent.state_dict(),
    'optimizer_state_dict': optimizer.state_dict(),
    'global_step': global_step,
    'episode': episode,
    'best_reward': best_reward,
}
torch.save(checkpoint, f"runs/train/exp{N}/checkpoint_{step}.pt")
```

支持断点续训，训练意外中断后可从最近 checkpoint 恢复。

### 2.7 YOLO 风格实验目录

仿照 YOLO 项目的实验管理方式：

```text
runs/
└── train/
    ├── exp1/    # 第 1 次训练 (已存在则自动递增至 exp2)
    ├── exp2/    # 第 2 次训练
    ├── exp3/
    └── ...
```

每次启动 `train.py` 自动创建递增编号的 `expN` 目录，所有 checkpoint、日志、TensorBoard 事件文件均存储于此。

## 3. 行为克隆预训练 — pretrain.py

### 3.1 动机

PPO 从零开始在仿真环境中训练，初期探索效率极低（大量随机动作导致频繁死亡）。行为克隆 (Behavior Cloning, BC) 使用采集的人类/脚本演示数据对策略网络进行监督预训练，使模型在 PPO 训练开始前已具备基本生存能力。

### 3.2 数据加载：mmap 惰性加载

训练数据以 `.npy` 格式存储，单文件可达数 GB。为避免一次性加载全部数据导致内存爆炸，采用 numpy 的 memory-mapped 文件模式：

```python
# 惰性加载，不会将数据读入内存
images = np.load("demonstrations/images.npy", mmap_mode='r')
labels = np.load("demonstrations/labels.npy", mmap_mode='r')
```

`mmap_mode='r'` 使得 numpy 仅在访问具体索引时才将对应数据页从磁盘加载到内存，而非整个文件。这确保了：
- 启动时 RAM 占用恒定（仅元数据开销）
- 无论训练多少 epoch，RAM 不会累积增长
- 对 SSD 友好的批量随机访问

### 3.3 数据集类与类别平衡

```python
class GameDataset(Dataset):
    def __init__(self, images, labels):
        self.images = images
        self.labels = labels

    def __getitem__(self, idx):
        return torch.from_numpy(self.images[idx]).float(), self.labels[idx]

    def __len__(self):
        return len(self.labels)
```

**类别不平衡问题**：演示数据中 `noop` (动作 4) 占比超过 70%，而跳跃 (0)、滑铲 (1) 占比极低，导致模型偏向预测 `noop`。

**解决方案：WeightedRandomSampler**：
```python
from torch.utils.data import WeightedRandomSampler

class_counts = np.bincount(labels)
class_weights = 1.0 / class_counts
sample_weights = class_weights[labels]

sampler = WeightedRandomSampler(
    weights=sample_weights,
    num_samples=len(sample_weights),
    replacement=True
)
```

每个样本被采样的概率与其所属类别的总出现次数成反比，使得稀有动作（跳、滑、切跑道）在训练中与高频动作（noop）获得均等的曝光机会。

### 3.4 训练/验证划分

80/20 分层划分：使用 `sklearn.model_selection.train_test_split` 配合 `stratify=labels`，确保训练集和验证集中各类别比例一致。

### 3.5 数据增强

```python
# 亮度抖动：模拟不同游戏场景明暗变化
brightness_factor = random.uniform(0.75, 1.25)
image = image * brightness_factor

# 随机平移：模拟截图时的轻微偏移
dx = random.randint(-4, 4)
dy = random.randint(-4, 4)
image = affine_transform(image, translate=(dx, dy))
```

增强操作应用于训练数据，验证数据保持原始状态。

### 3.6 优化器与调度器

| 组件 | 选择 | 参数 |
|------|------|------|
| 优化器 | AdamW | lr=3e-4, weight_decay=1e-4 |
| 学习率调度器 | ReduceLROnPlateau | mode='max', factor=0.5, patience=6 |
| 早停 | Early Stopping | patience=12 (连续 12 epoch 验证准确率不提升) |
| 损失函数 | CrossEntropyLoss | — |

### 3.7 评估指标

```python
# 整体准确率
val_acc = correct / total

# 各类别准确率
per_class_acc = {}
for action in range(5):
    mask = (val_labels == action)
    per_class_acc[action] = (pred[mask] == val_labels[mask]).float().mean()
```

训练日志同时记录整体验证准确率和每个动作类别的准确率，便于发现模型在某些动作上的偏见。

### 3.8 Checkpoint 保存策略

仅保存在验证集上达到最高 `val_acc` 的模型：
```python
if val_acc > best_val_acc:
    best_val_acc = val_acc
    torch.save(model.state_dict(), "pretrain_best.pt")
```

## 4. 仿真训练 — train_gym.py (当前主力)

### 4.1 概述

`train_gym.py` 是当前在用的主力训练代码，在上一章所述的 Gym 仿真环境中使用 PPO 算法训练。充分利用并行环境实现高效率采样。

### 4.2 网络架构：CNN+MLP 双塔结构

```text
                        ┌──────────────────────────────────────────┐
                        │              融合层 [3136+128]            │
                        └──────┬──────────────────────┬────────────┘
                               │                      │
              ┌────────────────┴────┐    ┌────────────┴─────────────┐
              │   CNN 塔 (视觉)      │    │   MLP 塔 (向量)           │
              │                     │    │                          │
              │ Conv(4→32,k8,s4)    │    │ obstacles(60d) +         │
              │   → ReLU            │    │ lane_onehot(3d) +        │
              │ Conv(32→64,k4,s2)   │    │ speed(1d)                │
              │   → ReLU            │    │   → 64d → ReLU           │
              │ Conv(64→64,k3,s1)   │    │   → 128d → ReLU          │
              │   → ReLU            │    │                          │
              │ Flatten → 3136d     │    │                          │
              └─────────────────────┘    └──────────────────────────┘
                               │                      │
              ┌────────────────┴────┐    ┌────────────┴─────────────┐
              │ 输入: obs (4,84,84) │    │ 输入: obstacles(10,6) +   │
              │                     │    │       lane(3) + speed(1)  │
              └─────────────────────┘    └──────────────────────────┘

                        ┌──────────────────────────────────────────┐
                        │              融合层 (3136+128)            │
                        └──────┬──────────────────────┬────────────┘
                               │                      │
                    ┌──────────┴──────┐    ┌──────────┴──────────┐
                    │ Actor: 256 → 5 │    │ Critic: 256 → 1     │
                    │ (动作概率分布)    │    │ (状态价值估计)        │
                    └─────────────────┘    └─────────────────────┘
```

**CNN 塔详细结构**：

| 层 | 输入通道 | 输出通道 | 卷积核 | 步长 | 输出尺寸 |
|----|---------|---------|--------|------|---------|
| Conv1 | 4 | 32 | 8x8 | 4 | (32, 20, 20) |
| Conv2 | 32 | 64 | 4x4 | 2 | (64, 9, 9) |
| Conv3 | 64 | 64 | 3x3 | 1 | (64, 7, 7) |
| Flatten | — | — | — | — | 3136 |

**MLP 塔详细结构**：

| 层 | 输入维度 | 输出维度 | 说明 |
|----|---------|---------|------|
| 展平 obstacles | (10, 6) | 60 | 10 行 x 6 特征 |
| lane 独热 | 3 | 3 | 当前跑道 |
| speed | 1 | 1 | 归一化速度 |
| 拼接 | 64 | — | 上述三者拼接 |
| FC1 | 64 | 128 | + ReLU |

**融合与输出**：

| 层 | 输入维度 | 输出维度 |
|----|---------|---------|
| 拼接 | 3136+128 | 3264 |
| Actor FC | 3264→256 | 5 (动作 logits) |
| Critic FC | 3264→256 | 1 (状态价值) |

### 4.3 PPO 训练配置

```python
# 环境配置
N_ENVS = 8                     # 8 个并行环境

# PPO 超参
LEARNING_RATE    = 3e-4        # 学习率
GAMMA            = 0.99        # 折扣因子
GAE_LAMBDA       = 0.95        # GAE λ 参数
CLIP_EPSILON     = 0.2         # PPO 裁剪 ε
ENTROPY_COEF     = 0.01        # 熵正则系数
CRITIC_COEF      = 0.5         # Critic 损失权重
MAX_GRAD_NORM    = 0.5         # 梯度裁剪最大值

# 收集与优化
ROLLOUT_STEPS    = 128         # 每个环境收集 128 步后优化
PPO_EPOCHS       = 4           # 每次优化遍历数据 4 轮
MINI_BATCH_SIZE  = 64          # 小批量大小

# 训练目标
TARGET_STEPS     = 5_000_000   # 总训练步数 (500 万步)
```

### 4.4 训练循环流程

```text
┌──────────────────────────────────────────────────────────────┐
│                  for step in range(TARGET_STEPS):             │
│                                                              │
│  1. 并行收集 128 步 × 8 环境 = 1024 组 (s, a, r, s', done)  │
│     ├─ 8 个环境同时执行 env.step(action)                     │
│     └─ 存入 RolloutBuffer                                    │
│                                                              │
│  2. 计算 GAE (Generalized Advantage Estimation)              │
│     ├─ 用 Critic 估计最后一帧价值                             │
│     └─ 回溯计算 δₜ = rₜ + γV(sₜ₊₁) - V(sₜ)                  │
│         Aₜ = Σ (γλ)ᵏ δₜ₊ₖ                                    │
│                                                              │
│  3. PPO 优化 (4 epochs × mini-batches)                      │
│     ├─ 从 buffer 随机抽取 mini-batch (size=64)               │
│     ├─ 计算新旧策略概率比 r = π_new(a|s) / π_old(a|s)        │
│     ├─ 计算裁剪损失 L_clip = min(r·A, clip(r,1-ε,1+ε)·A)    │
│     ├─ 加上 Critic 损失和价值熵                              │
│     └─ Adam 优化器更新参数                                    │
│                                                              │
│  4. 日志记录                                                  │
│     ├─ TensorBoard: reward, loss, entropy, etc.              │
│     └─ 终端: 当前步数、平均 reward、FPS                       │
│                                                              │
│  5. 每 N 步保存 checkpoint                                    │
│     └─ runs_gym/expN/checkpoint_{step}.pt                     │
│                                                              │
│  end for                                                     │
└──────────────────────────────────────────────────────────────┘
```

## 5. PPO 超参对比表

### 5.1 真机 vs 仿真超参对比

| 参数 | 真机 v1 (Nature CNN 128) | 真机 v2 (ResNet-18 320) | 仿真 (Dual-Tower) |
|------|:------:|:------:|:------:|
| 学习率 (lr) | 3e-4 | 3e-4 | 3e-4 |
| 折扣因子 (γ) | 0.99 | 0.99 | 0.99 |
| GAE λ | 0.95 | 0.95 | 0.95 |
| PPO 裁剪 ε | 0.2 | 0.2 | 0.2 |
| 熵正则系数 (ent_coef) | 0.02 | 0.02 | 0.01 |
| Critic 系数 (critic_coef) | 0.5 | 0.5 | 0.5 |
| 最大梯度范数 | 0.5 | 0.5 | 0.5 |
| rollout_steps | 512 | 256 | 128 |
| PPO epochs | 4 | 4 | 4 |
| mini_batch_size | 64 | 32 | 64 |
| n_envs | 1 | 1 | 8 |

### 5.2 参数差异分析

**ent_coef (0.02 → 0.01)**：仿真环境中熵系数从 0.02 降至 0.01。由于仿真环境 8 环境并行收集，样本多样性天然高于单环境真机训练，因此可以适当降低熵正则强度，让策略更快收敛而非过度探索。

**rollout_steps (512 → 256 → 128)**：步数递减反映了并行化带来的等效性。8 环境 × 128 步 = 1024 步总样本/轮，相比单环境 512 步/轮，总样本量翻倍但每环境步数更少，批内样本相关性更低。

**mini_batch_size (64 → 32 → 64)**：仿真恢复为 64。真机 v2 因 ResNet-18 参数量大（11M）且单环境收集样本少（256），需更小的 mini-batch 以提高梯度更新频率。仿真模型参数量约 0.5M，mini_batch=64 足够。

**n_envs (1 → 8)**：单环境真机受限于模拟器独占，无法并行。仿真环境纯计算，8 并行是 RTX 3060 单卡运行下 CPU-GPU 吞吐平衡的经验值。

## 6. 模型架构演进

| 版本 | 输入 | 网络架构 | 参数量 | 用途 |
|------|------|---------|--------|------|
| v0 | (15, 128, 128) | Nature CNN 3-layer Conv | ~1M | 真机初始实验 |
| v1 | (15, 320, 320) | ResNet-18 + Spatial Attention | ~11M | 真机升级版 |
| v2 | 14-dim vector | MLP 14→128→128 | ~5K | 仿真上界测试 (oracle) |
| v3 | (160, 160, 3) | CNN 3-layer Conv | ~100K | 仿真纯视觉 |
| v4 | Dict(obs + vector) | CNN+MLP Dual-Tower | ~500K | 仿真混合 (当前) |

### v0 — Nature CNN (初始版)

基于 Mnih et al. 2015 的 DQN 论文中 Nature CNN 架构，使用 3 层卷积处理 128x128 的 15 帧堆叠输入。~1M 参数量适合初期快速实验，但受限于输入分辨率和网络容量，在真机上表现有限。

### v1 — ResNet-18 + Spatial Attention (升级版)

将输入分辨率提升至 320x320，CNN 骨干替换为预训练的 ResNet-18。引入空间注意力模块 (Spatial Attention) 来聚焦游戏画面中的关键区域（跑道、障碍物、金币）。参数量 ~11M，训练时间成倍增加但是唯一在真机上展现一定泛化能力的版本。

### v2 — MLP Oracle (仿真上界)

直接使用 14 维完美向量特征（跑道、障碍物位置/类型、速度）作为输入，通过 3 层 MLP 学习策略。~5K 参数量即可快速收敛。此版本用于验证：当输入特征完美时，PPO 是否能够学会该任务（作为仿真环境的上界基准）。

### v3 — CNN 纯视觉 (仿真)

仅使用 (160, 160, 3) RGB 图像作为输入，与真实视觉更接近。训练速度慢于 v4，且收敛不稳定（策略对视觉扰动敏感），作为纯视觉基线。

### v4 — CNN+MLP 双塔 (当前主力)

结合 v2 (向量特征效率高) 和 v3 (视觉特征信息丰富) 的优势。CNN 塔处理 4 帧堆叠 84x84 俯视图，MLP 塔处理显式特征向量，融合后输入 Actor/Critic 头。~500K 参数量，在 GPU 上的推理和训练开销极低。
