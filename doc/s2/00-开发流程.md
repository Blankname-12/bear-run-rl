# 00-开发流程：从真机截屏到仿真环境

> S2 (第一新对话) | 2026-05-16 ~ 2026-05-20 | 完整开发叙事

---

## 阶段零：起点状态

本次对话开始时，项目已有一个可运行的真机 RL 训练系统：

- **环境**: 雷电模拟器运行熊大快跑 APK
- **截屏**: minicap (socket JPEG流, ~50ms/帧)
- **触控**: ADB exec-out Popen (~5ms 发送)
- **观测**: 4帧 × 128×128 RGB + 3赛道one-hot = 15通道
- **网络**: Nature CNN (3层卷积, ~1M参数)
- **算法**: PPO (clip ε=0.2, GAE λ=0.95)
- **奖励**: 存活+0.5, 死亡-10, 帧差法全局危险区-0.3
- **问题**: 训练了几千局，智能体仍然在 20 步暴毙

用户的核心诉求是"为什么纯视觉方案训练不出来"，希望我帮助诊断和改进。

## 阶段一：纯视觉方案诊断 (第1轮对话)

### 1.1 瓶颈分析

我完整阅读了 `config.py`、`agent.py`、`game_env.py`、`train.py`、`collect_data.py`、`fast_io.py`，识别出四个致命瓶颈：

1. **分辨率太低 (128×128)**: 游戏画面裁切后约 960×1500 → 缩放到 128×128 → 障碍物从 50-80px 变成 7-10px → Nature CNN 的 3 层 conv 根本"看不见"障碍物
2. **网络太浅**: 3 层卷积 (kernel 8→4→3, channel 32→64→64) 无法从像素学习高级语义
3. **奖励太稀疏**: 只有死亡时才给 -10，平时只有 +0.5 存活奖励，帧差法危险区检测噪声巨大
4. **minicap 不稳定**: 15fps 锁帧，且 x86_64 二进制在模拟器上频繁崩溃

### 1.2 提出的方案

我提出了五个方案，按优先级排序：
- 方案一(首选): 提升分辨率 + ResNet-18 + 帧差法障碍物定位
- 方案二: YOLO 障碍物检测辅助
- 方案三: 更好的奖励塑形
- 方案四: 增强行为克隆预训练
- 方案五: OCR 分数提取

用户选择了方案一，明确要求分辨率"越高越好"，不要 256，要 320+，CNN 改用 ResNet-18 级别。

### 1.3 关键决策：不用 YOLO

用户拒绝了 YOLO 辅助方案，理由是"轻量化 YOLO 推理速度跟不上动作响应"。这个决策是合理的——在实时游戏中，每增加 1ms 推理延迟就少 1ms 反应时间。最终采用"高分辨率让 CNN 自己看见障碍物 + 帧差法给奖励信号"的折衷方案。

## 阶段二：ResNet-18 升级实施 (第2-3轮对话)

### 2.1 config.py 改动

把 `FRAME_WIDTH/HEIGHT` 从 128 改为 320。这个数字的考量：
- 224 是 ResNet 标准输入但用户说"太低了"
- 256 被用户明确拒绝
- 384 时单条状态 15×384×384×4 = 8.8MB，ROLLOUT_STEPS=256 时需要 2.2GB 仅状态存储
- 320 是平衡点：单条 ~6MB，256 条 ~1.5GB

同步删除了旧的 DANGER_ZONE_* 配置（全局危险区），新增了 OBSTACLE_* 四个参数用于全分辨率帧差法。

奖励从三个信号（存活/死亡/危险区）扩展为五个（存活/死亡/正确闪避/漏闪避/前方安全）。

### 2.2 agent.py 重写

从 ~100 行的 Nature CNN 改为 ~300 行的自实现 ResNet-18：
- BasicBlock: 标准残差块 (conv3→BN→ReLU→conv3→BN → +shortcut)
- 4 个 layer (64→64→128→256→512)
- SpatialAttention 插入 layer3 之后（20×20 特征图，每个 cell 对应输入 16×16 区域，正好覆盖障碍物大小）
- RolloutBuffer 改为 uint8 存储（state×255→uint8，节省 75% 内存）

**为什么不直接用 torchvision 的 ResNet-18？** 因为第一层 conv 需要接受 15 通道输入（4帧×3RGB + 3赛道），而 torchvision 的 ResNet-18 第一层固定为 3 通道。

### 2.3 game_env.py 改动

这是最复杂的改动。核心思想：**用全分辨率帧差法替代粗粒度危险区全局均值**。

旧的 `_check_danger()` 只做一件事：裁出画面底部 1/3，算帧差均值，大于 8.0 就报警。问题：
- 不知道障碍物在哪条赛道
- 不知道障碍物是什么类型
- 光影变化/UI闪烁/角色动画都触发误报

新的 `_detect_obstacles()` 流程：
```
prev_cropped vs curr_cropped (全分辨率裁切区域)
  → cv2.absdiff → cv2.threshold → 形态学开运算(去噪)
  → findContours → 按面积过滤 → 按 x 坐标分赛道
  → 返回 [(lane_0idx, y_ratio, area), ...]
```

新增 `_compute_obstacle_reward()`：如果本赛道有障碍物且智能体做了任意闪避动作→+1.0；如果本赛道有障碍物但智能体 no-op→-2.0；如果无障碍且直行→+0.05。

**关键细节：换道后 2 帧冷却** (`_diff_cooldown=2`)。因为换道时整个屏幕会横向移动，帧差会有巨大的全屏变化，如果不冷却会被误判为障碍物。

### 2.4 此时系统的致命缺陷

改完代码后，面临一个尴尬局面：**新架构从未在真机上实际训练过，因为数据采集系统本身有 BUG**。

## 阶段三：数据采集 BUG 大爆发 (第4-12轮对话)

这是整个项目最痛苦的阶段。每次修改似乎都引入新问题，形成了"改一个 → 崩另一个 → 再改 → 再崩"的循环。

### 3.1 第一次尝试收集数据

用户运行 `collect_data.py`，报错：
```
OSError: [Errno 28] No space left on device
```

**根因**: 我把 memmap 的 capacity 设为 10000，在 320×320 float32 下需要 57.2GB 预分配，用户磁盘没这么多空间。

**修复**: capacity 降到 500，dtype 从 float32 改为 uint8（每条约 1.5MB，500 条 ≈ 768MB）。

### 3.2 数据采集时程序自动干预游戏

用户手动玩游戏时，程序突然自动点击了"取消"按钮，弹出暂停菜单。

**根因**: `GameEnv.observe()` 内部硬编码了死亡检测逻辑（`_heart_lost()` → `_verify_death()` → 自动点按钮）。这在训练模式下是必需的（agent 死了要自动重启），但在采集模式下会干预用户操作。

**修复**: 给 `observe()` 加 `check_death` 参数，采集时传 `False`。

### 3.3 仍然自动干预 + 截屏慢

用户报告 W/A/S/D 操控失败——按了键游戏没反应。日志显示每帧截屏耗时 2+ 秒，并且仍然有死亡检测的日志。

**根因有两个**：
1. 死亡检测虽然在 `observe()` 中被关了，但 `collect_data.py` 的主循环在收到 `done=True` 后仍然调用了 `env.restart_game()`——这是训练逻辑的残留
2. 截屏慢是因为 `_adb_screencap()` 有 4 种方法 × 3 次重试 = 最坏 30+ 秒/帧。minicap 崩溃后回退链触发了全部方法

**修复**: 
- 主循环中删除所有 `done` 检查和自动重启逻辑——用户自己按 R 重启
- `_adb_screencap()` 从 90 行精简到 23 行，只保留 `exec-out screencap -p` 一种方法，一次重试

### 3.4 截屏方法切换的混乱

minicap 崩溃后，我在 `config.py` 中尝试了三种截屏模式：

| 顺序 | 模式 | 结果 |
|------|------|------|
| 第一次 | `CAPTURE_MODE = "minicap"` | 崩溃，垃圾分辨率 70778880×125829120 |
| 第二次 | `CAPTURE_MODE = "window"` | 窗口只有 358×674，截图画质废掉 |
| 第三次 | `CAPTURE_MODE = "adb"` | 稳定，但需要精简回退链 |

**Window 模式为什么失败**: 雷电模拟器在屏幕上是一个小窗口（358×674），但游戏内部以 1080×1920 渲染。mss (DXGI) 捕获的是**窗口像素**，不是**游戏像素**。截出来的图只有 1/3 大小，所有坐标变换都错了。

### 3.5 异步截屏的尝试与放弃

为了让键盘不因截屏慢而卡顿，我尝试了后台线程截屏 + 主线程键盘的方案。但实际效果是：

- 后台线程 `observe()` 阻塞在 ADB 截屏（2+秒）
- 主线程 `wait_new_frame()` 忙等新帧产出（2+秒）
- 键盘轮询在忙等期间完全不执行
- 结果：异步没有带来任何好处，只是把阻塞换到了另一个地方

**结论**: 当生产者（截屏）比消费者（键盘）慢时，异步无意义。最终回到同步模型。

### 3.6 Ctrl+C 死循环

用户按 Ctrl+C 想退出训练，但程序反复打印"收到中断信号"却不退出。

**根因**: 
1. `signal_handler` 只设了 `interrupted=True`，但主线程当前可能阻塞在 `subprocess.run(adb screencap)` 中（C 扩展，无法被 Python 信号打断）
2. 内层循环退出后，post-episode 代码仍执行（统计 + 保存 + `restart_game()` 的 3.5 秒 sleep 链）
3. 用户在这 4+ 秒内连按 Ctrl+C，每次触发一次打印
4. `restart_game()` 成功执行后，游戏被重启，下一局开始——看起来像死循环

**修复**: 信号计数器（第一次设 flag，第二次 `os._exit(1)` 强制退出）；内层循环退出后立即 `if interrupted: break`；`restart_game()` 前加 `not interrupted` 保护。

### 3.7 数据采集最终方案

经过十轮反复修改后，`collect_data.py` 被简化到了极致：

```python
# 核心循环
key = read_keyboard()
if key in ACTIONS:
    save(state, raw_frame, action)          # 直接 tofile 写磁盘
    do_action(action)                        # ADB 发送滑动
    time.sleep(0.15)                         # 等动作生效
    raw = screencap()                        # adb exec-out screencap -p
    processed = preprocess_frame(raw)        # 裁剪+缩放
    push_frame(processed)                    # 更新帧栈
    state = get_state()                      # 构建模型输入
```

- 没有 GameEnv
- 没有 observe()
- 没有 fast_io / minicap
- 没有死亡检测
- 没有线程
- 没有回退链
- 总共约 150 行

每步约 400ms（ADB 截屏占大头），虽然慢但稳定运行。

## 阶段四：APK 逆向尝试 (第13-15轮对话)

### 4.1 动机

用户提出："能不能逆向解包熊大快跑，保留干净的游戏逻辑，在 Python 里复刻一个环境？"

这个想法的核心洞察是：**与其跟模拟器/ADB/minicap 死磕，不如把游戏逻辑提取出来，直接写一个 Python 版游戏，训练速度可以快 1000 倍**。

### 4.2 APK 获取与分析

APK 路径在模拟器的 `/storage/emulated/0/MT2/apks/熊出没之熊大快跑_4.6.0.apk`。ADB 拉取时遇到中文路径问题（Bash/PS 的路径翻译差异），最终在模拟器内复制为 `/sdcard/xd.apk` 后拉取成功。文件 351MB，15 秒拉完。

解包确认：
- `libil2cpp.so` + `global-metadata.dat` → **Unity IL2CPP 模式**
- 无 `Assembly-CSharp.dll`（IL2CPP 编译后 C# 变成了 C++ native code）
- 架构: arm64-v8a + armeabi-v7a

### 4.3 Il2CppDumper 下载失败

IL2CPP 反编译需要 Il2CppDumper 工具（将 `global-metadata.dat` + `libil2cpp.so` → 可读的 C# 伪代码）。但我尝试了所有下载途径：
- GitHub 直连 → 被墙
- ghproxy.com → 连接失败
- ghfast.top → 返回 9 字节（空响应）
- gitee → 不存在此仓库
- pip install il2cppdumper → 无此包

全部失败。

### 4.4 自写 Metadata 解析器

放弃外部工具，直接写 `parse_metadata.py` 解析 `global-metadata.dat` 的二进制格式。文件结构：
- Magic: 0xFAB11BAF (byte-swapped), Version: 31
- 字符串索引: 120,376 个条目（每个条目 = int32 offset + int32 length，指向字符串数据区）
- 字符串数据区: 367,316 字节

成功提取了全部 120K+ 字符串，过滤出游戏相关字符串后得到关键信息：
- **主 DLL**: BearGame.dll, GameHelper.dll, TrackMaterialType.dll
- **核心类**: PlayerControl, ScoreRaceManager, ScoreGameUI
- **动作**: Jump, Slide, openMoveType
- **障碍物**: openMechanism1-4（4种机制）, ObstacleID
- **赛道**: Mphalane, Labohlane, Diphalane（3种赛道类型名）
- **状态**: bIsDead, curSpeed, CurrentLane, CoinCount

### 4.5 UE5 开源项目发现

用户在 GitHub 上找到了 `XiongDaRun-main`——一个 UE5.6 项目，疑似熊大快跑的简化实现。核心 C++ 类：

- `ARunner`: 3 赛道(0/1/2), LaneWidth=270, ForwardRunSpeed=1.0, MoveLeft/MoveRight
- `AFloorSegment`: 1000 单位长, 3 spawn 行, SpawnItems() 含 20% 障碍物 + 40% 金币概率
- `AObstacleBase`: Box Collision → OnOverlap → Runner->Die()
- `AWordGenerate`: GameMode, 无限生成 segment

**关键发现**: UE5 源码只有左右移动，**没有跳跃/滑铲逻辑**！真实的熊大快跑（从 APK 字符串推断）有完整的 Jump/Slide 动作和 4 种障碍物机制。

## 阶段五：Gym 仿真环境构建 (第16-17轮对话)

### 5.1 信息来源交叉验证

仿真环境 `xiongda_env.py` 的设计基于三个来源的交叉验证：

| 参数 | UE5 源码 | APK 字符串 | 实际游玩 |
|------|---------|-----------|---------|
| 3 赛道 | ✅ LaneWidth=270 | ✅ CurrentLane | ✅ |
| 左右移动 | ✅ MoveLeft/Right | ✅ | ✅ |
| 跳跃 | ❌ 无 | ✅ Jump | ✅ |
| 滑铲 | ❌ 无 | ✅ Slide | ✅ |
| 障碍物类型 | ❌ 单一 ObstacleBase | ✅ openMechanism1-4 | ✅ 3种可见 |
| 金币 | ✅ Coin 40%概率 | ✅ CoinCount | ✅ |
| 速度递增 | ❌ 恒定 1.0 | ✅ curSpeed | ✅ |
| 死亡判定 | ✅ 碰撞即死 | ✅ PlayDieAnim | ✅ |

对于 UE5 源码缺失的跳跃/滑铲，通过实际游玩观察推断：
- 起跳到落地约 0.5s
- 滑铲到起身约 0.5s
- 换道耗时约 0.15s

### 5.2 环境设计

观测空间采用 Dict 而非纯图像，融合了结构化信息：
```
{
  "obs":        (4, 84, 84)  # 4帧灰度俯视图
  "lane":       Discrete(3)  # 当前赛道
  "obstacles":  (10, 6)      # 前方10个障碍物 [距离,赛道,类型,是否在本赛道,...]
  "speed":      (1,)         # 归一化速度
}
```

**为什么是 84×84 而不是 320×320？** 因为这是仿真环境，障碍物位置精确已知（通过 obstacles 向量直接提供），不需要 CNN 从高分辨率像素中识别。84×84 的俯视图只提供"大概的空间布局感"。

### 5.3 速度对比

这才是仿真环境的真正价值：

| 指标 | 真机 | 仿真 |
|------|------|------|
| 单步耗时 | 400ms~2000ms | <0.1ms |
| 并行环境数 | 1 | 8 (可更多) |
| 每秒总步数 | ~2 | ~10,000+ |
| 100万步 | ~6天 | ~2分钟 |

**速度提升约 4000-5000 倍**。

### 5.4 训练脚本

`train_gym.py` 使用 CNN+MLP 双塔架构处理 Dict 观测：
- CNN 塔: 处理 (4,84,84) 图像 → 3136 维特征
- MLP 塔: 处理障碍物向量 + 赛道 + 速度 → 128 维特征
- 融合: 3136+128=3264 → Actor(256→5) / Critic(256→1)

8 个并行环境，ROLLOUT_STEPS=128，目标 500 万步。

### 5.5 可视化

`visualize.py` 使用 OpenCV 做了 3D 透视渲染：梯形赛道、虚线分隔、障碍物按类型显示不同形状（低矮=横条、高处=竖条、地面=方块）、金币为黄点、玩家为底部圆圈带跳跃/滑铲动画。

最初尝试用 pygame，但 pygame 在 Python 3.12 上因 `pkgutil.ImpImporter` 被移除而崩溃。换成 OpenCV 后稳定运行。

## 关键转折点总结

| # | 转折点 | 触发原因 | 决策 | 影响 |
|---|--------|---------|------|------|
| 1 | 放弃 Nature CNN 128×128 | 障碍物视觉不可见 | 升级到 ResNet-18 320×320 | 架构复杂度大幅增加 |
| 2 | 放弃 minicap | 模拟器上频繁崩溃 | 改用 ADB exec-out | 帧率从 15fps 降到 2fps |
| 3 | 放弃 Window DXGI | 窗口太小截图画质废 | 改用 ADB exec-out | 帧率进一步降低但稳定 |
| 4 | 放弃异步截屏 | 截屏本身是瓶颈，异步不解决问题 | 回归同步模型 | 代码从 300 行降到 150 行 |
| 5 | 放弃完整 APK 反编译 | Il2CppDumper 被墙 | 自写 metadata 解析器 + UE5 源码 | 获得足够信息构建 Gym 环境 |
| 6 | 放弃真机训练 | 截屏/触控/死亡检测不稳定，训练太慢 | 构建纯 Python Gym 环境 | 训练速度提升 4000x |
| 7 | 放弃 pygame 可视化 | Python 3.12 不兼容 | 改用 OpenCV | 稳定但交互性差一些 |

## 未完成/已知局限

1. **Gym 环境不完全还原真实游戏**: 跳跃/滑铲逻辑是基于推断而非反编译，障碍物生成概率来自 UE5 源码（可能不同于真实游戏）
2. **仿真→真机迁移未验证**: 在仿真环境训练的模型能否直接用于真实游戏，尚未测试
3. **数据采集仍需人工**: 没有实现自动录制（需要人工按键），数据量有限
4. **Il2CppDumper 仍未获取**: 完整反编译仍是被墙状态，无法 100% 确认游戏逻辑
