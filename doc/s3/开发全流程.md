## 阶段 1：尝试切换到高速截屏方案

### 1.1 切换到 scrcpy（失败）

**想法**：scrcpy 是 Google 官方推荐的 Android 屏幕镜像工具，使用 GPU 硬件编码 H.264，通过 ADB socket 传输。minicap 不稳定，scrcpy 应该更可靠。

**实现**：在 `fast_io.py` 新增 `ScrcpyCapture` 类（~200 行），流程：
```
1. 下载 scrcpy-server-v2.4.jar → bin/scrcpy/
2. adb push 到 /data/local/tmp/
3. adb forward tcp:port → localabstract:scrcpy
4. adb shell CLASSPATH=... app_process ... Server 2.4 ...
5. socket 连接 → 收 H.264 裸流
6. ffmpeg 子进程解码 H.264 → RGB
```

**遇到的第一个问题**：`get_scrcpy()` 中 `connect()` 抛异常后全局变量 `_scrcpy` 已指向半初始化对象，后续调用不再重试，反复报 `'NoneType' object has no attribute 'stdin'`。

修复：`connect()` 成功后才赋值 `_scrcpy = cap`，失败时 `_scrcpy` 保持 None。

**遇到的第二个问题**：scrcpy 服务端正常启动（diagnostic 脚本确认：socket 可连、进程存活），但 ffmpeg 解码管道永久不输出帧。

加了 `-fflags nobuffer -flags low_delay -probesize 32` 等低延迟参数，仍然无效。

怀疑 ffmpeg 在等 IDR 关键帧——scrcpy 的 `send_device_meta=false` 使流直出 NAL 单元，ffmpeg 要等到第一个 IDR 帧才初始化解码器。

**尝试的替代方案**：
- PyAV 进程内解码 → `av.CodecContext.create('h264','r')` parse→decode → 同样无输出
- `adb exec-out screenrecord --output-format=h264 -` → 能解码，但仅 1 FPS

**结论**：scrcpy 服务端完全正常，但模拟器端的编码/传输是整个管道的瓶颈，不是 PC 解码端的问题。放弃 scrcpy。

**学到的**：
- 不要用全局变量缓存有状态连接对象（初始化失败要重置为 None）
- H.264 解码的一个陷阱：裸流需要 IDR 帧才能开始解码
- `send_device_meta=false` 时流不包含设备元数据，别尝试解析 header

### 1.2 修复 minicap（半成功）

**思路**：minicap 虽然不稳定但是唯一能在隐藏窗口下高速截屏的方案，与其放弃不如修复。

**问题分析**（用户指出三个根因）：版本不匹配、系统资源冲突、服务管理不当。

**修复 1 — 版本匹配**：改进 `_find_local_binaries()`，严格匹配 ABI 和 SDK，加入 `x86_64` 候选，多级 SDK 回退，失败时列出所有可用文件方便诊断。

**修复 2 — 资源控制**：`nice -n 10` 低优先级，`LD_LIBRARY_PATH` 确保 so 找得到。

**修复 3 — 服务管理**：
```python
MAX_FAILURES = 5         # 连续失败阈值 → 触发完整重建
RESTART_INTERVAL = 120   # 每 120 秒主动重启（防内存泄漏）

_health_check()          # 定时检查 pidof minicap
_reconnect()             # 完整重建连接
_recv_exact(..., timeout=3.0)  # 每条 recv 带超时
get_frame() 外包裹 try/except + 失败计数 + 自动重建
```

**修复 4 — 头解析跳过**：minicap 的 24 字节 Banner 返回垃圾分辨率 `70778880×125829120`，直接跳过，用 `config.SCREEN_WIDTH/HEIGHT`。

**同时的配置优化**：
- `CAPTURE_MODE = "minicap"`
- `_heart_lost(raw, fast=True)` 单帧检测（避免 3 帧截屏 = 300ms 额外等待）
- 去掉 `-r` FPS 限制，让 minicap 全速跑

**结果**：minicap 稳定性明显改善，但仍偶发崩溃。作为备选方案保留。

### 1.3 尝试 ADB raw 格式（失败）

**想法**：`screencap -p` 输出 PNG，模拟器端 PNG 编码可能很慢。不用 `-p`，输出 raw 像素格式，跳过编码。

```python
def _adb_screencap_raw():
    r = subprocess.run(["adb", "exec-out", "screencap"], capture_output=True)
    w, h, fmt = struct.unpack_from("<III", r.stdout, 0)
    pixels = r.stdout[12:12 + w * h * 4]
    img = np.frombuffer(pixels, dtype=np.uint8).reshape((h, w, 4))
    return np.ascontiguousarray(img[:, :, :3])

# 测试: 最快 651ms, 中位 1007ms, 平均 1001ms, 1 FPS
```

**结论**：跳过 PNG 编码没省时间——跳过编码后 raw 数据是 1080×1920×4 = 8.3MB，ADB 传输这么大 raw 数据反而比 PNG 压缩后慢。ADB 通道本身就是瓶颈。

**学到的**：
- `np.frombuffer` 返回只读视图 → 加 `.copy()` 才能被 OpenCV 修改
- `img[:,:,:3]` 切片后数组不连续 → `np.ascontiguousarray()` 确保 C-contiguous
- 这两个是项目中的高频坑

### 1.4 发现拼写错误

用户反馈"速度慢"，但 config 里写的是 `"window"` 应该很快才对。排查发现 `config.py` 第 26 行是 `CAPTURE_MODE = "windows"`（多了 s）。

`"windows"` 不匹配任何已知模式（`"scrcpy"`, `"minicap"`, `"window"`），`capture_frame()` 全不命中，走到 ADB 兜底 → 1 FPS。所以感觉慢。

修复：`"windows"` → `"window"`，切回 mss 桌面捕获（17ms/帧，58 FPS）。

**结论**：**window (mss) 是最快最稳的方案**，唯一缺点是窗口不能最小化。

## 阶段 2：分析训练失败原因

用户已经训练了 18+ 个实验，但智能体始终学不会闪避。本阶段系统性地分析为什么。

### 2.1 分析 exp22（移除帧差法后的版本）

```
总轮数: 276
平均步数: 92
平均奖励: -31.06
PPO 更新: ~99 次
```

步数分布：69% 活过 100 步，17% 在 40 步内暴毙。

趋势：初期步数 89 → 后期步数 92，奖励 -31.55 → -29.71。**几乎没有改善。**

**诊断**：智能体学会了两极行为——
- "直跑" → 活 ~115 步（前 50 秒障碍物少）
- "探索" → 随机滑 → 大概率暴毙

每次探索失败，PPO 强化"别动"。然后熵系数又逼它探索，再失败。死循环。

**根本矛盾**：纯生存信号无法支撑障碍物闪避学习。智能体不知道"这一滑避开了障碍物"还是"这一滑纯粹多余"——两个都得到同样的 +0.2 存活奖励。

### 2.2 分析 exp40（参数调炸的版本）

用户将 `REWARD_HEART_LOST` 从 -0.5 提到 -10，`STEP_INTERVAL` 从 0.5 拉到 1.2。

```
总轮数: 855
平均步数: 19
平均奖励: -18.64
99.9% 暴毙（854/855 局 <40 步）
仅 1 局活到 1290 步（第 14 局异常值）
```

步数分位数：p50=17 步。一半的局在 20 秒内死亡。

趋势：初期(31步) → 中期(17步) → 后期(18步)。**在变差。**

奖励分位数 80% 集中在 [-18.6, -17.6]，极窄——智能体学会了把奖励稳定在这个区间：早死。活久反而可能扣更多（红心丢失 -10×n），所以最优策略是**早死早超生**。

**三条致命组合**：
1. `REWARD_HEART_LOST = -10` → 红心丢一帧 -10，死前两步 -20，是死亡惩罚的两倍
2. `STEP_INTERVAL = 1.2` → 1.2 秒动一次，障碍物出现到碰撞才 1-2 秒，反应窗口太窄
3. 预训练 82% no_op → PPO 探索暴毙 → 收敛到"别动"→ 死循环

**学到的**：
- 惩罚信号不能压倒正信号——否则智能体会走"少犯错"而非"做对事"
- 纯生存奖励不够——必须有时反馈（闪避成功 +1，撞了 -5）
- 动作间隔不能太长——要小于障碍物的反应窗口
- 预训练的 no_op bias 配合稀疏奖励 = 死局

## 阶段 3：奖励函数重构

基于 exp22/exp40 的分析，对奖励系统进行了重构。

### 3.1 移除帧差法

删除了以下代码：
- `_detect_obstacles()` — 帧差法障碍物检测（噪声过大）
- `_compute_obstacle_reward()` — 基于帧差法的奖励（有致命 bug：任意非 no_op 拿 +1.0）
- `REWARD_AVOID`, `REWARD_MISS`, `REWARD_CLEAR` — 三个障碍物相关配置
- `OBSTACLE_*` — 四个障碍物检测阈值
- `_prev_cropped`, `_diff_cooldown` — 帧差状态变量

### 3.2 保留四个可靠信号

```python
REWARD_ALIVE = 0.2        # 存活（每步）
REWARD_DEATH = -10.0      # 死亡
REWARD_HEART_LOST = -0.5  # 红心丢失（每帧堆叠）
REWARD_BOUNDARY = -0.5    # 撞墙（左车道左滑/右车道右滑）
```

### 3.3 边界惩罚

用户提出"左车道往左滑、右车道往右滑应该扣分"。之前是只不给存活奖励（0.0），差距太小。改为扣 -0.5（后来改到 -2.0）。

### 3.4 暖机改为红心检测

原来用固定 9 秒暖机：
```python
if time.time() < self._warmup_deadline:
    return next_state, 0.0, False
```

改为等红心首次出现：
```python
if not self._heart_seen:
    heart_px = _detect_heart_pixels(raw)
    if heart_px >= threshold:
        self._heart_seen = True  # 暖机结束
    else:
        return next_state, 0.0, False
```

这样做的好处：红心不出现（加载/死亡/广告期间）不训练，确保训练数据全是游戏画面。

### 3.5 心跳检测改为单帧

原来 `_heart_lost(raw, fast=False)` 做 3 帧多数投票，每秒额外 2 次截屏。改为 `fast=True` 单帧检测。配合 `HEART_DEATH_FRAMES=2` 足够可靠。

## 阶段 4：设想过"读游戏内存"方案

### 4.1 内存 Gym 环境

实现了 `memory_reader.py`（ADB root 读 `/proc/pid/mem`）和 `gym_env.py`（Gym 环境适配），观测改为 10 维向量（从内存读障碍物位置/类型/赛道/分数），而非 320×320 像素。

"用 Cheat Engine 找地址"麻烦 → 那不如自己写一个游戏，状态全在自己手里。这个想法直接催生了仿真方案。

## 阶段 5：仿真跑酷游戏开发

### 5.1 第一版：简单 2D 游戏

最初的 `simple_game.py` 是一个简陋的顶部视角游戏，3 条垂直线代表赛道，彩色方块代表障碍物。

用户反馈："可是我觉得这个就很简单了啊，还有比较作为课设吗"——太简单不够格。

### 5.2 第二版：加入 pygame 渲染

加了伪 3D 透视效果。画出路面、车道线、天空。但初始版本很粗糙——三条路不明显，障碍物就是彩色矩形。

**三个渲染问题**：
1. 不可见字符 U+200B 导致语法错误 → 删除重写
2. `pygame.font.SysFont` 崩溃 → 改用 `pygame.font.Font(None, size)`
3. 三条车道不明显 → 重新设计路面

### 5.3 渲染迭代（多轮优化）

**路面重绘**：从逐行 line drawing 改为填充多边形（性能更好）：
- 绿色草地 → 灰色路面梯形 → 2 条白粗实线（车道分隔） → 3 条灰虚线（车道中心线） → 红色护栏

**深度感**：
- 天空渐变（从深蓝到浅蓝）
- 远山轮廓
- 路边树木（4 棵/侧，不同远近不同大小）
- 大气雾化（`_fog_blend`，远处物体颜色褪向天蓝）

**障碍物 3D 化**：
- 跨栏：红色横杆 + 两个棕色支柱
- 矮栏：贴地橙色块 + 黑色危险条纹
- 路障：半个紫色矩形（左路障挡左边，右路障挡右边）

**玩家**：影子椭圆 + 两条腿 + 身体矩形（带高光）+ 头 + 头发

**动态效果**：金币光晕、速度线（高速时）、Game Over 遮罩

### 5.4 最终游戏参数

```python
3 赛道，5 动作（跳/滑/左/右/直跑）
4 种障碍物，需要 3 种闪避方式
碰撞即死（精确 lane + type + y 判定，无延迟）
速度从 2.5 渐进递增到 10.0（课程学习）
金币随机生成，收集 +1.0 奖励 +10 分
```

## 阶段 6：训练系统构建

### 6.1 MLP 网络（向量模式，~19K 参数）

```
14 → Linear(128)+ReLU → Linear(128)+ReLU → Actor(5) / Critic(1)
```

### 6.2 CNN 网络（像素模式，~500K 参数）

```
(160,160,3) → Conv(32,k8,s4) → Conv(64,k4,s2) → Conv(64,k3,s1)
→ Flatten(16384) → FC(256)+ReLU → Actor(5) / Critic(1)
```

自动检测模型类型：检查 checkpoint 的 key 名是否含 "conv"。

### 6.3 PPO + GAE

```python
lr=1e-3, γ=0.99, λ=0.95, clip=0.2, epochs=4, batch=128, ent_coef=0.02
```

### 6.4 训练命令

```bash
python train_simple.py                  # MLP, ~1 分钟
python train_simple.py --visual         # CNN, ~10 分钟
python train_simple.py --compare        # 两者对比
```
