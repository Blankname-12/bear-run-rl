# 会话 3 完整记录

> 日期: 2026-05-18 | 总耗时: ~4小时 | 对话轮次: ~100轮

## 做什么 / 为什么做

本项目目标是训练一个强化学习智能体玩 Temple Run 风格跑酷游戏。经过前两个会话的尝试（Android 模拟器 + ADB 截屏方案），遇到了严重的 I/O 瓶颈和训练效果问题。本会话的核心任务是：

1. 排查并解决截屏方案的性能和稳定性问题
2. 分析训练失败的原因（为什么几千局还在暴毙？）
3. 在真机方案走不通的情况下，设计替代方案
4. 实现可用的训练流水线

## 本会话做了什么

### 截屏方案排查（约占 30% 时间）
- 切到 scrcpy 并完整调通服务端 → 发现 ffmpeg 解码管道从未输出帧
- 尝试 PyAV 替代 ffmpeg → 同样解不出帧
- 回到 minicap 并修复三个根因（版本/资源/服务管理）
- 测试 ADB raw 格式（跳过 PNG 编码）→ 仍然只有 1 FPS
- 发现 `CAPTURE_MODE = "windows"` 拼写错误导致降级到 ADB → 修正
- 结论：**window (mss) 是唯一可用方案**，需要窗口保持可见

### 训练分析（约占 20% 时间）
- 分析 exp22：276 局，奖励曲线完全平坦，PPO 更新 99 次但未学到闪避
- 分析 exp40：855 局，99.9% 暴毙，平均 19 步，没有学习趋势
- 诊断根因：纯生存信号无法支撑障碍物闪避学习 → 需要即时动作反馈

### 奖励重构（约占 15% 时间）
- 移除帧差法障碍物奖励（噪声过大）
- 将暖机从固定时间改为红心检测
- 简化 `step()` 逻辑
- 发现 `REWARD_HEART_LOST=-10` 过于激进导致"早死早超生"

### 仿真环境（约占 35% 时间，核心产出）
- 设计并实现纯 Python 跑酷游戏（伪 3D 渲染）
- 实现 MLP + CNN 双模式 PPO 训练
- 实现 AI 试玩可视化
- 多次迭代渲染细节（路面/车道线/树木/玩家/雾化等）
