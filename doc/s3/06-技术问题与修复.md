## 1. OpenCV 中文路径写文件失败

### 症状
```python
cv2.imwrite("D:\\一些word\\...\\calibration.png", img)
# 返回 True, 但文件不存在
```

### 原因
`cv2.imwrite` 底层调用 C 标准库的 fopen，Windows 上的 fopen 使用 ANSI 编码，无法处理含中文字符的路径。

### 修复
```python
# 改为: imencode + ndarray.tofile
cv2.imencode('.png', cv2.cvtColor(img, cv2.COLOR_RGB2BGR))[1].tofile(path)
# imencode 返回 (success, buffer), buffer.tofile() 使用 Python 的文件 I/O, 支持 Unicode
```

### 影响文件
- `calibration.py` line 158
- `collect_data.py` line 343
- 项目中所有 `cv2.imwrite` 调用

## 2. np.frombuffer 只读数组

### 症状
```python
img = np.frombuffer(pixels, dtype=np.uint8).reshape((h, w, 4))
cv2.rectangle(img[:,:,:3], ...)  # 报错
# cv2.error: NumPy array marked as readonly
```

### 原因
`np.frombuffer` 返回的数组是底层 bytes 对象的视图，不拥有自己的内存，标记为 WRITEABLE=False。

### 修复方案演进
- v1: `img = ...reshape((h,w,4)).copy()` → 产生整个 RGBA 的副本
- v2: `img[:,:,:3].copy()` → 只产生 RGB 切片的副本（但切片本身可能不连续）
- v3: `np.ascontiguousarray(img[:,:,:3])` → 最终方案，既生成副本又确保内存连续

### 影响
`game_env.py` `_adb_screencap_raw()` 函数

## 3. numpy 切片后内存不连续

### 症状
```python
img = np.frombuffer(pixels, dtype=np.uint8).reshape((1920, 1080, 4)).copy()
rgb = img[:, :, :3]
# cv2.rectangle(rgb, ...) 报错
# "Layout of the output array img is incompatible with cv::Mat"
```

### 原因
对于 shape (1920, 1080, 4) C-contiguous 数组，stride = (4320, 4, 1)。
切片 `[:,:, :3]` 后 shape = (1920, 1080, 3)，但 stride 仍为 (4320, 4, 1)。
C-contiguous 要求 stride[1] == shape[2] × stride[2] = 3×1 = 3，但实际 stride[1] = 4。
→ 数组不是 C-contiguous，OpenCV 无法处理。

### 修复
```python
rgb = np.ascontiguousarray(img[:, :, :3])
```
`ascontiguousarray` 总是返回 C-contiguous 的副本。

## 4. pygame.SysFont 中文 Windows 崩溃

### 症状
```python
pygame.font.SysFont("consolas", 18)
# pygame 内部扫系统字体目录 → 遇到中文用户名路径
# → initsysfonts_win32 → splitext(font) → font 是 int → TypeError
```

### 原因
pygame 2.6.1 的 `initsysfonts_win32` 函数在遍历 Windows 字体注册表时，遇到了非字符串类型的条目。中文用户名下的路径编码导致 `splitext` 收到 int 而非 str。

### 修复
```python
# 不用 SysFont, 用内置默认字体
self._font = pygame.font.Font(None, 20)
self._big_font = pygame.font.Font(None, 50)
```
`Font(None, size)` 使用 pygame 内置的默认字体，不扫描系统字体。

## 5. 不可见 Unicode 字符 U+200B (Zero Width Space)

### 症状
```
File "simple_game.py", line 221
    ​0: (255, 100, 100),  # 跨栏 - 红色
    ^
SyntaxError: invalid non-printable character U+200B
```

### 原因
Write 工具生成字典键 `0`, `1`, `2`, `3` 时混入了零宽空格字符（U+200B，UTF-8 编码为 `\xe2\x80\x8b`）。

### 修复
重写整个 `_COLORS` 字典，确保键为纯整数。

### 检测方法
```python
with open('simple_game.py', 'rb') as f:
    data = f.read()
    if b'\xe2\x80\x8b' in data:
        print('U+200B found!')
```

### 症状


### 修复
改为 `"window"`。

## 7. scrcpy 设备元数据解析乱码

### 症状
```
[Scrcpy] 已连接: 2648gB)h@<y<"h5e...乱码..., 31330×51213
```

### 原因
scrcpy 启动参数 `send_device_meta=false` 时不发送设备名/分辨率头，流数据直接是 H.264 NAL 单元。代码仍尝试解析 `[1B name_len][name][2B w][2B h]`，把 H.264 裸流字节当成了元数据。

### 修复
跳过 header 解析，直接用 `config.SCREEN_WIDTH=1080`、`config.SCREEN_HEIGHT=1920`。

### 同样的问题
- minicap 的 24 字节 Banner 解析也返回垃圾分辨率 (70778880×125829120)
- 修复: minicap 也跳过 width/height 解析，只取 PID 确认连接

## 8. 心跳检测 3 帧截屏拖慢训练

### 症状
每步耗时远超预期。

### 原因
`_heart_lost(raw, fast=False)` 内部:
```python
for _ in range(2):
    time.sleep(0.03)
    lost_count += 1 if _frame_no_heart(get_screen_numpy()) else 0
```
- 每分钟调用 2 次额外的 `get_screen_numpy()`
- minicap 模式每分钟额外 2×50ms = 100ms
- 加上主截屏 50ms = 150ms
- 加上 STEP_INTERVAL sleep = 总步时 >1s

### 修复
```python
# step() 中改用单帧检测
heart_lost = self._heart_lost(raw, fast=True)
```
`HEART_DEATH_FRAMES=2` 下，单帧检测已足够可靠。

## 9. subprocess text=True 崩溃

### 症状
```
UnicodeDecodeError: 'gbk' codec can't decode byte 0x80
```
发生在 `adb push` 命令的输出捕获。

### 原因
`subprocess.run(..., capture_output=True, text=True)` 使用系统默认编码（Windows 上为 GBK）解码 stdout/stderr。`adb push` 输出包含二进制进度条，无法用 GBK 解码。错误发生在 subprocess 内部的读线程中，即使主线程 try/except 也捕获不到。

### 修复
```python
# 始终用 text=False (bytes)
r = subprocess.run(["adb", ...], capture_output=True)
# 需要文本时手动解码
text = r.stdout.decode("utf-8", errors="replace")
```

### 影响
`test_scrcpy.py` 的 `adb()` 辅助函数

## 11. 模型类型不匹配

### 症状

## 13. 协程/线程中的 ADB 并发问题

### 症状
多个线程/异步任务同时使用 ADB 时偶发失败。

### 原因
ADB server 是单线程模型，多连接共享状态。`adb forward`、`adb shell` 等操作间可能有竞争。

### 修复
- 避免 ADB 操作的并发调用
- 关键操作（如 minicap 启动/关闭）加 `time.sleep()` 等待

## 14. 训练时 loss 列始终为空

### 症状
CSV 的 loss 列全是空白。

### 原因
`train.py` 中 `logger.log(..., None, ...)` 硬编码了 loss=None。`agent.update()` 的返回值被丢弃。

### 修复
在仿真训练脚本中正确记录 loss：
```python
loss = agent.update() or 0.0
writer.writerow([..., f"{loss:.4f}", ...])
```
