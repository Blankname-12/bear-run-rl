# 截屏方案深度分析

## 背景

训练 RL 智能体需要持续获取游戏画面。本项目运行环境为雷电模拟器（LDPlayer，Android 9，x86_64），通过 ADB 与 PC 通信。截屏方案的选择直接影响训练可行性和效率。

理想要求：
- 帧率 >10 FPS（每帧 <100ms），越高越好
- 稳定运行数小时不崩溃
- 模拟器窗口可最小化（不占用桌面空间）

本会话系统性地测试了全部可行方案。

---

## 方案一: Windows DXGI 桌面捕获（mss）

### 原理
使用 `mss` 库调用 Windows DirectX Graphics Infrastructure (DXGI) 桌面复制 API，直接抓取屏幕像素。`game_env.py` 中通过 `win32gui.EnumWindows` 自动检测模拟器窗口客户区坐标，mss 按坐标区域截取。

### 实现代码路径
```
get_screen_numpy()                      # game_env.py:214
  → fast_io.capture_frame()            # fast_io.py:308
    → _get_screen_window()             # game_env.py:96
      → _get_window_rect()             # game_env.py:81
        → _detect_emulator_window()    # game_env.py:51
      → mss.mss().grab(rect)           # 截取窗口区域
```

### 窗口检测逻辑
```python
def _detect_emulator_window():
    # 枚举所有顶层窗口
    win32gui.EnumWindows(callback, None)
    # 匹配关键词: ["雷电","LDPlayer","MEmu","夜神","Nox","BlueStacks","MuMu","Android","Emulator"]
    # 返回窗口客户区屏幕坐标和尺寸
```

### 实际性能
- 单帧耗时: **~17ms**
- 帧率: **58 FPS**（受显示器刷新率限制）
- 截取分辨率: 取决于窗口大小（校准会计算缩放比）

### 限制
- **窗口必须可见且非最小化**：DXGI 只能捕获屏幕上实际渲染的像素
- 缩放因子需校准确认：窗口大小 ≠ 游戏原生分辨率(1080×1920)，需要 `_capture_scale` 换算

### 结论
性能最优，但窗口不能隐藏。与用户需求（训练时用电脑做其他事）矛盾。

## 方案二: scrcpy（设备端硬件编码 + socket 流）

### 原理
scrcpy 是 Google 官方推荐的 Android 屏幕镜像工具。其服务端（Java JAR）运行在设备中，通过 Android MediaCodec API 进行 GPU 加速 H.264 编码，再通过 ADB 端口转发将视频流传到 PC。PC 端需解码 H.264 恢复 RGB 图像。

### 服务端启动流程
```bash
# 1. 推送 JAR 到设备
adb push scrcpy-server.jar /data/local/tmp/

# 2. 转发端口
adb forward tcp:<local_port> localabstract:scrcpy

# 3. 启动服务端（app_process 是 Android 的 Java 运行时）
adb shell "CLASSPATH=/data/local/tmp/scrcpy-server.jar \\
           app_process / com.genymobile.scrcpy.Server 2.4 \\
           tunnel_forward=true audio=false control=false \\
           send_device_meta=false send_frame_meta=false \\
           send_dummy_byte=false max_size=1920 bit_rate=8000000"
```

### 参数说明
- `tunnel_forward=true`: 使用 ADB 端口转发（不走独立 TCP）
- `audio=false`: 不传输音频
- `control=false`: 不需要反向控制
- `send_device_meta=false`: 不发送设备名/分辨率（流直接是纯 H.264）
- `send_frame_meta=false`: 不发送 PTS 等帧元数据
- `send_dummy_byte=false`: 不发送虚拟字节标记
- `max_size=1920`: 最大分辨率边，维持 1080×1920 不变
- `bit_rate=8000000`: 8 Mbps 码率

### 诊断脚本 (`test_scrcpy.py`)
```python
# 第一步: 确认服务端进程存活
adb shell "pidof app_process"
# 输出: shell 17809 ... app_process → 确认存活

# 第二步: 测试 socket 连接
sock.connect(("127.0.0.1", port))
header = sock.recv(64)
# 成功收到数据 → socket 通道正常
```

### 诊断结果
- ✅ 服务端 push 成功（JAR 69,007 bytes）
- ✅ `app_process` 进程启动成功（pidof 确认）
- ✅ ADB forward + socket 连接成功
- ✅ 收到初始字节

### PC 端解码尝试

#### 子进程 ffmpeg 方案
```python
# 管道: socket → ffmpeg stdin → ffmpeg stdout → numpy
ffmpeg_cmd = [
    "ffmpeg",
    "-fflags", "nobuffer",       # 禁用输入缓冲
    "-flags", "low_delay",       # 低延迟模式
    "-probesize", "32",          # 最小探测数据量
    "-analyzeduration", "0",     # 不分析时长
    "-f", "h264",                # 输入格式: 裸 H.264
    "-i", "pipe:0",              # 从 stdin 读取
    "-f", "rawvideo",            # 输出格式: 裸像素
    "-pix_fmt", "rgb24",         # 像素格式: RGB 24bit
    "-an", "-sn",                # 无音频/字幕
    "-nostats", "-loglevel", "error",
    "-"                          # 输出到 stdout
]
proc = subprocess.Popen(ffmpeg_cmd, stdin=PIPE, stdout=PIPE, stderr=PIPE)
```

**症状**: `connect()` 中喂入初始 socket 数据后，`proc.stdout.read(frame_size)` 永远不返回。ffmpeg 进程未崩溃（`poll()` 返回 None），stdin 写入成功，但 pipe 中无帧输出。

**分析**: ffmpeg 的 `-f h264` 输入格式要求流以 SPS/PPS/IDR 帧头开始才能初始化解码器。虽然加了 `-fflags nobuffer -probesize 32` 减少缓冲，但 ffmpeg 内部仍等待 IDR 关键帧。scrcpy 的 `send_device_meta=false` 模式从第一个 NAL 单元开始流，ffmpeg 在碰到 IDR 帧前不会输出任何解码帧。

**失败**: 5 秒超时后报 `"ffmpeg 在 5 秒内未输出帧"`

#### PyAV 进程内方案
```python
import av
codec = av.CodecContext.create('h264', 'r')
buf = bytearray()
# 从 socket 收数据 → parse 出 packets → decode 成 frames
packets = codec.parse(buf)
for packet in packets:
    frames = codec.decode(packet)
    for frame in frames:
        img = frame.to_ndarray(format='rgb24')
```

**症状**: 同样无输出。`codec.parse()` 返回空列表，或 `codec.decode()` 不产帧。

**分析**: PyAV 内部也是 ffmpeg 库，同样受 IDR 帧约束。

#### screenrecord 替代方案
```bash
# Android 内置录屏命令，输出标准 H.264（含 SPS/PPS/IDR 头）
adb exec-out screenrecord --output-format=h264 --size=1080x1920 --bit-rate=8000000 -
```
- 能收到 8 帧（1080×1920），确认解码可通
- 但实际帧率仅 **~1 FPS**，和 ADB 一样慢

### 结论
- scrcpy 服务端完全正常（socket 可连，进程可查）
- H.264 解码在等待 IDR 帧时阻塞，这是所有 H.264 解码器的固有限制
- screenrecord 虽能解码但帧率太低
- **模拟器端的视频编码/ADB 传输是整个管道的瓶颈，不是 PC 解码端的问题**

## 方案三: minicap（设备端 JPEG 编码 + socket 流）

### 原理
minicap 是 Android 屏幕捕获工具，通过直接读取 SurfaceFlinger 帧缓冲获取画面，编码为 JPEG 后通过 socket 发送。与 scrcpy 不同，minicap 使用软件编码（CPU），不需要 Android MediaCodec。

### 本会话对 minicap 的修复

#### 问题 1: 版本不匹配
```
设备: x86_64, SDK 28
bin/minicap/ 中的文件: x86_64/minicap + android-28/x86_64/minicap.so
```
修复: 改进 `_find_local_binaries()` 的搜索逻辑，加入 `x86_64` 候选，多级 SDK 回退，失败时列出所有可用文件。

#### 问题 2: 头解析错误
```python
# 旧代码试图解析 24 字节 Banner
header = self._recv_exact(24)
(version, _, pid, real_w, real_h, ...) = struct.unpack("<BBHIIIII", header)
# 输出: PID=15904, 70778880×125829120  ← 明显不对
```
原因: minicap 版本间的 Banner 结构不同。修复: 跳过宽度/高度解析，只用 `config.SCREEN_WIDTH`、`config.SCREEN_HEIGHT`。

#### 问题 3: 资源耗尽
```
minicap 进程被 Android low-memory killer 杀掉
→ socket 断连 → 死等 → 训练卡死
```
修复:
- `nice -n 10` 降低进程优先级
- `LD_LIBRARY_PATH=/data/local/tmp:/system/lib64` 确保 so 文件找到
- 去掉 `-r` 帧率限制（让 minicap 全速跑）
- `-S` 标志: 消费慢时丢弃旧帧

#### 问题 4: 服务管理缺失
```python
class MinicapCapture:
    MAX_FAILURES = 5            # 连续失败触发重建
    RESTART_INTERVAL = 120      # 每120秒主动重启（防内存泄漏）

    def _health_check(self):
        # 1. 定时主动重启
        if time.time() - self._last_restart > self.RESTART_INTERVAL:
            self._reconnect()
        # 2. 进程存活检查
        r = subprocess.run(["adb", "shell", "pidof", "minicap"], ...)
        if not r.stdout.strip():
            self._reconnect()

    def _recv_exact(self, n, timeout=3.0):
        # 每条 recv 有超时上限，防止永久阻塞
        deadline = time.time() + timeout
        ...

    def get_frame(self):
        try:
            return self._get_frame_inner()
        except Exception as e:
            self._failures += 1
            if self._failures >= self.MAX_FAILURES:
                self._reconnect()  # 自动重建整个连接
            raise
```
修复: 心跳检查 + 超时保护 + 故障计数 + 自动重建。

#### 问题 5: get_frame 死等
旧代码在 `latest_jpeg is None` 时调用 `self._recv_exact(4)` + `self._recv_exact(frame_size)`，这两个调用无超时限制。一旦 socket 断开，永久阻塞。

修复: 所有 `_recv_exact` 调用增加 timeout 参数（默认 3 秒）。

### 性能
- 单帧: ~50ms（取决于模拟器 CPU 性能）
- 帧率: ~10-20 FPS
- 稳定性: 改进后明显好转，但仍偶发崩溃

### 最终参数
```bash
# 全速截屏, -S 丢弃慢消费帧, nice 低优先级
adb shell "LD_LIBRARY_PATH=/data/local/tmp:/system/lib64 \\
           nice -n 10 nohup /data/local/tmp/minicap \\
           -P 1080x1920@1080x1920/0 -S >/dev/null 2>&1 &"
```

## 方案四: ADB screencap（设备端 PNG / raw 输出）

### ADB PNG 模式
```bash
adb exec-out screencap -p
# 输出: PNG 格式的屏幕截图
```
- 耗时: ~680ms
- 瓶颈: 模拟器内部 PNG 编码 + ADB 传输全屏数据

### ADB Raw 模式（本会话新实现）
```bash
adb exec-out screencap
# 输出: [4B width][4B height][4B format][raw pixels RGBA]
```
```python
def _adb_screencap_raw():
    r = subprocess.run(["adb", "exec-out", "screencap"], capture_output=True)
    w, h, fmt = struct.unpack_from("<III", r.stdout, 0)
    pixels = r.stdout[12:12 + w * h * 4]
    img = np.frombuffer(pixels, dtype=np.uint8).reshape((h, w, 4))
    return np.ascontiguousarray(img[:, :, :3])  # RGBA → RGB
```

性能:
- 最快: 651ms | 中位: 1007ms | 最慢: 1361ms | 平均: **1001ms**
- 帧率: **~1 FPS**
- 瓶颈: skips PNG 编码，但 ADB 传输 1080×1920×4 = 8.3MB raw 数据仍然慢

### 结论
- 跳过 PNG 编码有一定提升（1000ms vs 680ms，实际 raw 反而更慢，因为 raw 数据量是 PNG 的几十倍）
- ADB 通道本身的吞吐量有限，模拟器端的 framebuffer 读取也有延迟
- **不适用于需要实时交互的 RL 训练**

## 综合对比

| 方案 | 每帧耗时 | 有效帧率 | 窗口可隐藏 | 稳定性 | 额外依赖 | 最终 |
|------|---------|---------|-----------|--------|---------|------|
| window (mss) | 17ms | 58 FPS | ❌ 必须可见 | ⭐⭐⭐⭐⭐ | mss, pywin32 | ✅ 选用 |
| minicap | 50ms | 10-20 FPS | ✅ | ⭐⭐⭐ (修复后) | minicap 二进制 | 🟡 备选 |
| scrcpy | 30-50ms | 理论 20-30 FPS | ✅ | ❌ 从未解码成功 | ffmpeg/PyAV, JAR | ❌ 放弃 |
| ADB raw | 1000ms | 1 FPS | ✅ | ⭐⭐⭐⭐⭐ | 无 | ❌ 太慢 |
| ADB PNG | 680ms | 1.5 FPS | ✅ | ⭐⭐⭐⭐⭐ | 无 | ❌ 太慢 |
| screenrecord | 1000ms | 1 FPS | ✅ | ⭐⭐⭐⭐ | PyAV/ffmpeg | ❌ 太慢 |

## 最终决策

**window (mss) 是唯一性能可用的方案。**代价是模拟器窗口必须保持可见。

要隐藏窗口可用替代方案：
1. Win+Ctrl+D 创建第二虚拟桌面，模拟器放桌面 2，工作在桌面 1
2. 休息时间/晚上挂机训练

## 心跳检测对截屏速度的影响

`step()` 中的心跳检测原本是 3 帧多数投票：
```python
def _heart_lost(self, raw_frame, fast=False):
    if fast:
        lost = _frame_no_heart(raw_frame)  # 单帧
    else:
        lost_count = 1 if _frame_no_heart(raw_frame) else 0
        for _ in range(2):
            time.sleep(0.03)
            lost_count += 1 if _frame_no_heart(get_screen_numpy()) else 0
        lost = lost_count >= 2
```

`fast=False` 时每分钟额外调用 2 次 `get_screen_numpy()` = 每分钟额外 2×50ms（minicap）/ 2×17ms（window）。

本会话修改为 `step()` 统一使用 `fast=True`，将 3 次截屏降为 1 次。配合已降低的 `HEART_DEATH_FRAMES=2`，单帧检测足够可靠。

## CAPTURE_MODE 拼写错误事件

`config.py` 行 26 被误写为 `CAPTURE_MODE = "windows"`（多了 `s`）。这不是合法值。`capture_frame()` 的匹配逻辑：
```python
if config.CAPTURE_MODE == "scrcpy": ...    # "windows" ≠ "scrcpy", 跳过
if config.CAPTURE_MODE == "minicap": ...   # "windows" ≠ "minicap", 跳过
if config.CAPTURE_MODE in ("window", ...):  # "windows" ∉ 集合, 跳过
# 全不匹配, 走到 ADB 回退
```

结果: 尽管设置了 "window"，实际跑的是 ADB 回退（1 FPS），导致"感觉速度明显慢"。
