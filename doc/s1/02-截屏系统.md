# 02 — 截屏系统

## 问题定义

熊大快跑运行在雷电模拟器（Android 9.0, x86_64）上，Python 控制端需要实时获取游戏画面。截屏是整个 RL 系统的基础——没有低延迟的画面输入，智能体无法做出及时决策。

**核心需求**：
1. 延迟 < 50ms（保证 ≥20 FPS 的操作频率）
2. 原生 1080×1920 分辨率（避免坐标缩放导致检测偏差）
3. 稳定可靠（支持数小时的训练/采集会话）

---

## 方案一：ADB screencap（初始方案 · 已废弃）

### 实现方式

```python
def _adb_screencap():
    # 方法1: shell stdout
    r = subprocess.run(["adb", "-s", serial, "shell", "screencap", "-p"],
                       capture_output=True, timeout=10)
    if r.returncode == 0 and len(r.stdout) > 200:
        return r.stdout.replace(b'\r\n', b'\n')

    # 方法2: 文件中转
    for png_path in ["/data/local/tmp/_cap.png", "/sdcard/_cap.png"]:
        subprocess.run(["adb", "-s", serial, "shell", "screencap", "-p", png_path], check=True)
        subprocess.run(["adb", "-s", serial, "pull", png_path, local_path], check=True)
        with open(local_path, "rb") as f:
            return f.read()
```

**数据流**：
```
Device: screencap → PNG 编码 → /sdcard/xxx.png (2.2MB)
Network: adb pull → TCP 传输 2.2MB → PC 本地文件
Python:  open + PIL decode → numpy array
```

### 性能实测

```
ADB screencap -p (stdout): 682ms/帧
ADB screencap -p + pull:   750ms+/帧
```

每帧延迟约 700ms，相当于 1.4 FPS。对于需要实时交互的游戏 AI，完全不可用。

### 延迟分解

| 环节 | 耗时 | 占比 |
|------|------|------|
| ADB 命令往返 | ~50ms | 7% |
| 设备端 PNG 编码（1080×1920） | ~300ms | 44% |
| TCP 传输 2.2MB | ~200ms | 29% |
| Python PIL decode | ~50ms | 7% |
| 其他开销 | ~80ms | 12% |

瓶颈在于设备端 **PNG 编码**和 **TCP 传输**。2.2MB 的数据量对于 ADB 通道来说太大了。

### 尝试的优化

1. `adb exec-out screencap -p`：绕过 PTY，节省 ~50ms → 仍 ~630ms
2. 原始格式（无 -p 标志，输出 RGBA 裸数据）：跳过 PNG 编码，但 1080×1920×4 = 8.3MB 需要传输，TCP 传输时间翻倍 → 总延迟反增到 ~800ms
3. 多路径尝试（/data/local/tmp, /sdcard, /tmp）：文件系统差异可忽略

结论：ADB screencap 方案在物理上不可行，TCP 通道带宽和 PNG 编码开销无法接受。

## 方案二：Windows DXGI 桌面捕获（过渡方案 · 已废弃）

### 原理

Python 运行在 Windows 上，模拟器窗口在桌面上可见。使用 `mss` 库调用 Windows DXGI (Desktop Duplication API) 直接捕获 GPU 显存中的模拟器窗口像素。

```python
import mss
import numpy as np

def _get_screen_window():
    rect = _get_window_rect()  # 自动检测模拟器窗口客户区
    with mss.mss() as sct:
        img = sct.grab(rect)
        return np.array(img)[:, :, [2, 1, 0]]  # BGRA → RGB
```

**数据流**：
```
GPU 显存 → DXGI Duplication → 系统内存 → numpy array
（全程在本地，不经过网络）
```

### 自动窗口检测

```python
def _detect_emulator_window():
    """通过 win32gui 枚举窗口，匹配标题关键词"""
    result = []
    def callback(hwnd, extra):
        if win32gui.IsWindowVisible(hwnd):
            title = win32gui.GetWindowText(hwnd)
            if any(kw in title for kw in config.EMULATOR_KEYWORDS):
                left, top = win32gui.ClientToScreen(hwnd, (0, 0))
                right, bottom = win32gui.ClientToScreen(
                    hwnd, win32gui.GetClientRect(hwnd)[2:])
                result.append((left, top, right-left, bottom-top, title))
    win32gui.EnumWindows(callback, None)
    return result[0] if result else None
```

关键词列表：`["雷电", "LDPlayer", "MEmu", "夜神", "Nox", "BlueStacks", "MuMu", "Android", "Emulator"]`

使用 `ClientToScreen` + `GetClientRect` 而非 `GetWindowRect`，排除标题栏和边框，只捕获客户区（游戏渲染区域）。

### 性能实测

```
首次捕获: 241ms (DXGI 初始化)
后续捕获: 14-29ms/帧
平均: ~17ms/帧 = ~60 FPS
```

### 致命的缺陷

#### 1. 分辨率不匹配
模拟器窗口大小由用户拖动决定（如 633×1078），而游戏原生分辨率为 1080×1920。所有坐标（裁剪区域、红心区域、按钮位置）都必须乘以缩放比：

```python
_capture_scale = (window_width / 1080, window_height / 1920)
# 例如: (633/1080, 1078/1920) = (0.586, 0.562)
```

红心区域从 120×50 缩到 70×28 像素，HSV 检测精度大幅下降。

#### 2. 色彩偏差
DXGI 捕获的像素值是经过 GPU 渲染管线处理后的最终显示色，与 ADB 直接获取的 Android SurfaceFlinger 原始帧相比，饱和度和亮度有系统性偏移。红心 HSV 检测阈值（S≥60, V≥80）在 DXGI 截图上几乎检测不到红色。

#### 3. 侧边栏偏移
雷电模拟器左侧有工具栏，游戏画面不在客户区 (0,0) 位置，需要额外配置 `GAME_OFFSET_X`。

#### 4. 窗口可见性约束
模拟器窗口不能最小化或被其他窗口完全遮挡，DD API 无法捕获被遮挡的窗口。

#### 5. 截图范围包含 UI
dxgi 截图包括了整个模拟器窗口（含工具栏、状态栏），游戏区域仅占一部分。裁剪时必须精确配置 GAME_CROP。

**结论**：DXGI 捕获性能极好（~17ms），但坐标偏差和色彩问题导致红心检测失效，不适合需要精确坐标的 RL 场景。

## 方案三：minicap（最终方案）

### 原理

minicap 是 STF (Smartphone Test Farm) 项目的核心组件之一，在 Android 设备上以原生代码（C++）运行：
1. 通过 Android SurfaceFlinger API 直接获取屏幕内容
2. 使用 libjpeg-turbo 进行 JPEG 压缩
3. 通过 Unix Domain Socket 或 TCP 推送帧流

与 ADB screencap 的关键区别：
- screencap 每次调用都重新初始化 → minicap 一次初始化，持续推送
- screencap 输出 PNG → minicap 输出 JPEG（体积 1/15）
- screencap 通过 ADB 传输 → minicap 通过 socket 直连

### 二进制文件部署

#### 文件来源
从 https://github.com/DeviceFarmer/minicap/releases 下载 v2.7.3（约 4.4MB zip）。

解压后结构：
```
minicap-2.7.3/
  libs/{abi}/minicap              ← 可执行文件（各 ABI）
  jni/minicap-shared/aosp/libs/
    android-{sdk}/{abi}/minicap.so ← SDK 专用共享库
```

LDPlayer 需要：
- ABI: `x86_64`（也可用 `x86` 32位版本，x86_64 向下兼容）
- SDK: `28`（对应 Android 9.0）

#### 目录组织
```
bin/minicap/
  x86_64/minicap           ← 来自 libs/x86_64/minicap (~789KB)
  android-28/x86_64/
    minicap.so             ← 来自 jni/.../android-28/x86_64/minicap.so (~28KB)
```

#### 推送与安装
```bash
# 推送文件到临时目录
adb push bin/minicap/x86_64/minicap /data/local/tmp/minicap
adb push bin/minicap/android-28/x86_64/minicap.so /data/local/tmp/minicap.so
adb shell chmod 755 /data/local/tmp/minicap

# Android 9+ 禁用了 LD_LIBRARY_PATH
# 必须把 minicap.so 放到系统库路径
adb shell su -c 'mount -o rw,remount /'
adb shell su -c 'cp /data/local/tmp/minicap.so /system/lib64/minicap.so'
adb shell su -c 'chmod 644 /system/lib64/minicap.so'
```

**为什么不能只用 LD_LIBRARY_PATH**：
Android 9 (API 28) 引入了 `android:useLegacyPackaging` 限制，linker 对非系统路径的 so 加载做了限制。尝试 `LD_LIBRARY_PATH=/data/local/tmp /data/local/tmp/minicap ...` 会报：
```
CANNOT LINK EXECUTABLE "/data/local/tmp/minicap": library "minicap.so" not found
```

必须 root 后挂载 /system 为可写，复制到系统路径。

#### 启动命令
```bash
adb shell "killall -9 minicap 2>/dev/null; \
  nohup /data/local/tmp/minicap -P 1080x1920@1080x1920/0 -r 15 -S \
  >/dev/null 2>&1 &"
```

**参数详解**：

| 参数 | 说明 |
|------|------|
| `-P 1080x1920@1080x1920/0` | 显示投影：实际分辨率@虚拟分辨率/旋转角度。1080×1920 竖屏，0° 不旋转 |
| `-r 15` | **帧率限制 15fps**。这是解决触控延迟的关键参数 |
| `-S` | 消费者读取慢时跳过旧帧，防止缓冲区堆积 |
| `> /dev/null 2>&1 &` | 后台运行，丢弃日志输出 |
| `nohup` | 防止 hangup 信号导致进程退出 |

**`-s` 参数的误读**（重要踩坑记录）：
minicap 帮助中 `-s` 的描述是：
```
-s: Take a screenshot and output it to stdout. Needs -P.
```

这是**单次截图模式**——截一帧输出到 stdout 然后退出。不加 `-s` 才是 **socket 服务端模式**——持续监听抽象 Unix socket `@minicap` 并推送帧流。早期误以为 `-s` = socket，反复尝试导致 minicap 启动后立即退出。

### Socket 协议实现

#### ADB 转发
```python
local_port = random.randint(20000, 50000)
subprocess.run(["adb", "-s", serial, "forward",
    f"tcp:{local_port}", "localabstract:minicap"], check=True)
```

ADB forward 将本地 TCP 端口映射到设备上的抽象 Unix socket `@minicap`。

#### 全局头解析（24 bytes, little-endian）

| 偏移 | 字节 | 字段 | 示例值 |
|------|------|------|--------|
| 0 | 1 | version | 1 |
| 1 | 1 | reserved | 0 |
| 2 | 2 | pid (uint16) | 14280 |
| 4 | 4 | display_width (uint32) | 1080 |
| 8 | 4 | display_height (uint32) | 1920 |
| 12 | 4 | display_orientation (uint32) | 0 (0°) |
| 16 | 4 | output_orientation (uint32) | 0 |
| 20 | 4 | quirk_flags (int32/uint32) | 0 |

```python
header = sock.recv(24)
ver, _, pid, w, h, dor, oor, flags = struct.unpack("<BBHIIIII", header)
print(f"PID={pid}, {w}x{h}, orient={dor}°")
```

注意：第 5 个 4 字节字段可能是 int32 或 uint32，取决于 minicap 版本。实测用 `IIII`(unsigned) 可正确解析。

#### 帧流读取
```python
def get_frame(self):
    """读取最新一帧：排空缓冲区旧帧，只保留最后一帧"""
    self._sock.settimeout(0.002)  # 2ms 超时
    latest_jpeg = None

    # 非阻塞排空所有积压帧（只读 JPEG 字节，不解码）
    while True:
        try:
            # 读4字节帧大小
            size_bytes = b""
            while len(size_bytes) < 4:
                size_bytes += self._sock.recv(4 - len(size_bytes))
            frame_size = struct.unpack("<I", size_bytes)[0]

            # 读JPEG数据（~150KB）
            jpeg = b""
            while len(jpeg) < frame_size:
                jpeg += self._sock.recv(frame_size - len(jpeg))

            latest_jpeg = jpeg  # 只保留字节，不在此解码
        except (socket.timeout, BlockingIOError):
            break  # 缓冲区已空
        except Exception:
            break

    self._sock.settimeout(5.0)

    # 只解码最后保留的那一帧
    if latest_jpeg is None:
        # 缓冲区空的，阻塞等一帧
        size_bytes = self._recv_exact(4)
        latest_jpeg = self._recv_exact(struct.unpack("<I", size_bytes)[0])

    return np.array(Image.open(io.BytesIO(latest_jpeg)).convert("RGB"))
```

**为什么要 drain 旧帧**：minicap 持续推送，如果两次读取间隔 200ms，socket 缓冲区积累了 ~12 帧旧画面。直接读会拿到 12 帧前的画面，导致感知延迟。

**为什么 drain 时不解码**：每帧 JPEG decode 约 10ms，12 帧 = 120ms。drain 时只读原始字节丢弃，最后一次性解码，节省 110ms。

### 性能实测

```
首帧连接: ~2800ms (启动服务端+forward+socket connect+read header)
后续帧:   14-18ms/帧 @ 15fps
分辨率:   1080×1920 原生 (uint8 RGB)
JPEG大小: ~150KB/帧 (比 PNG 的 2.2MB 小 93%)
```

minicap 15fps 限速下，端到端帧率约 **20 FPS**（含截屏+预处理+检测），足以支持实时 RL 训练和数据采集。

### 帧率限制与触控延迟的关键关联

**这是本项目最重要的性能发现之一**。

**问题现象**：用户按下 W 键后约 1 秒游戏才有反应。

**排查过程**：
1. 怀疑 Popen 延迟 → 实测 ~5ms，排除
2. 怀疑 observe 阻塞 → 实测 ~20ms，排除
3. 怀疑 Python 代码逻辑 → 加时间戳，排除
4. 最终发现：minicap 不限制帧率时默认以最大速度推送（实测 ~60fps）

**根因分析**：
```
minicap 60fps → ~150KB/帧 × 60 = 9MB/s JPEG 流
ADB forward 通过同一个 ADB 通道传输
触控指令 (adb exec-out input swipe ...) 也在同一 ADB 通道
9MB/s 几乎占满 ADB 通道带宽
→ 触控指令在 ADB 服务端队列中排队
→ 用户按键后需等 ~1 秒触控才到达设备
```

**解决方案**：`-r 15` 限制 15fps
```
minicap 15fps → ~150KB/帧 × 15 ≈ 2.25MB/s
ADB 通道大部分空闲 → 触控指令即刻送达
```

这是用截屏帧率换取触控响应速度的经典 trade-off。15fps 对 RL 训练完全足够。

### 健壮性设计

minicap 可能因各种原因崩溃（模拟器重启、内存不足、协议错误），需要自动恢复：

```python
def capture_frame():
    """截屏入口：minicap → DXGI → ADB 三级回退"""
    if config.CAPTURE_MODE == "minicap":
        try:
            cap = get_capture()
            return cap.get_frame()
        except FileNotFoundError as e:
            print(f"[FastIO] minicap 二进制未找到: {e}")
            print("[FastIO] 回退到 Windows 桌面捕获")
        except Exception as e:
            print(f"[FastIO] minicap 异常: {e}")
            print("[FastIO] 回退...")

    # 回退到 DXGI
    if config.CAPTURE_MODE in ("window", "minicap"):
        from game_env import _get_screen_window
        frame = _get_screen_window()
        if frame is not None:
            return frame

    # 最终回退到 ADB
    from game_env import _adb_screencap
    png_bytes = _adb_screencap()
    return np.array(Image.open(io.BytesIO(png_bytes)).convert("RGB"))
```

三级回退确保即使 minicap 出问题，系统也能降级运行。

### 在游戏和训练中的表现

- **minicap 原生 1080×1920**：红心 HSV 检测精度高，按钮坐标无偏差，死亡判定准确
- **`-r 15` 帧率限制**：ADB 通道畅通，触控延迟降至可接受范围
- **drain 不解码优化**：每次 get_frame 耗时从 ~135ms 降到 ~20ms
- **可靠性**：在数小时训练中稳定运行，偶发 minicap 崩溃会触发自动重启
