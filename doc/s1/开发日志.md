## 阶段 1：截屏系统演进

### 1.1 初始状态：ADB screencap（682ms/帧）

**现状**：`game_env.py` 中 `_adb_screencap()` 通过 ADB shell 截图。

**性能实测**：
```
adb shell screencap -p (stdout):     682ms
adb exec-out screencap -p (无PTY):   630ms
adb shell screencap (无-p, 原始格式): 800ms+
```

**结论**：延迟在 600-800ms 范围，约 1.5 FPS，不可接受。

### 1.2 第一次尝试：Windows DXGI 桌面捕获

**动机**：跳过 ADB，直接从 Windows GPU 缓冲区捕获模拟器窗口。

**实现**：
- 安装 `mss` + `pywin32`
- `_get_screen_window()`：mss.grab() + BGRA→RGB
- `_detect_emulator_window()`：win32gui 枚举窗口，匹配"雷电"等关键词
- 使用 `ClientToScreen` + `GetClientRect`（排除标题栏）

**性能**：首帧 241ms（DXGI 初始化），后续 14-29ms/帧。极快。

**问题发现**：
1. 分辨率不匹配：窗口 633×1078 ≠ 1080×1920
2. 色彩偏差：DXGI 截图比 ADB 截图偏淡
3. 坐标偏移：雷电侧边栏导致游戏画面右移
4. 窗口必须可见

**尝试修复**：添加 `_capture_scale`（窗口/原生比例），所有坐标乘以缩放比。但 HSV 检测的 S_MIN/V_MIN 仍需针对 DXGI 截图放宽（60→40, 80→50）。

**部分成功，但引入新问题**：红心检测在 DXGI 截图上不稳定，且在切换回原生方案后忘记恢复阈值。

### 1.3 最终方案：minicap

**动机**：需要原生 1080×1920 分辨率 + 低延迟。minicap 是 STF 项目的 Android 原生截屏工具。

**部署过程**：
1. 从 DeviceFarmer/minicap releases 下载 v2.7.3
2. 组织目录：`bin/minicap/x86_64/minicap` + `bin/minicap/android-28/x86_64/minicap.so`
3. 推送 + root remount + 安装 so 到 /system/lib64/

**踩坑：`-s` 参数误解**
- 误以为 `-s` = socket 模式
- 实际 `-s` = "Take a screenshot and output it to stdout"（单帧模式）
- 不带 `-s` 才是 socket 服务端模式（监听 @minicap）

**踩坑：LD_LIBRARY_PATH 被忽略**
- Android 9+ 安全限制
- 必须 root + remount /system + 复制 so 到 /system/lib64/

**踩坑：60fps 占满 ADB 通道**
- 不限帧率时 minicap 推送 ~9MB/s JPEG 流
- 触控指令排队 1+ 秒
- **这是最关键的发现**：截屏和触控共享 ADB 带宽，必须平衡
- 修复：`-r 15` 限制 15fps

**踩坑：drain 时解码每帧**
- socket 缓冲区积压 ~12 帧旧画面
- 每帧 PIL decode ~10ms → drain 耗时 120ms
- 修复：drain 只读字节不解码，最后一次性 decode

**最终性能**：~17ms/帧（15fps），原生 1080×1920。

## 阶段 2：触控系统演进

### 2.1 初始状态：subprocess.run + adb shell（182ms）

```python
subprocess.run(["adb", "shell", "input", "swipe", ...], capture_output=True)
```
每次等待 ADB 返回，总耗时 ~182ms。

### 2.2 第一次优化：exec-out（55ms）

```python
subprocess.run(["adb", "exec-out", "input", "swipe", ...], capture_output=True)
```
绕过 PTY，耗时降至 ~55ms。但仍然是阻塞等待。

### 2.3 持久化 Shell 尝试（失败）

**设计**：维持长期 `adb shell` 进程，stdin 管道持续发命令。

**失败原因**：Windows 上，`adb shell` 的 stdin 非 TTY 时立即退出。尝试 `while read cmd; do eval $cmd; done`、`adb shell -T` 等均失败。

**结论**：放弃持久化方案。

### 2.4 最终方案：Popen + exec-out（~5ms）

```python
subprocess.Popen(["adb", "exec-out", "input", "swipe", ...],
    stdout=DEVNULL, stderr=DEVNULL)
```
fire-and-forget，Python 端 ~5ms 返回。封装为 FastTouch 类（swipe/tap 异步，tap_sync 同步）。

### SWIPE_DURATION_MS 调参

| 尝试 | 结果 |
|------|------|
| 50ms | 被 Android 识别为点击 |
| 80ms | 仍有误判 |
| 300ms | 延迟太高，操作拖泥带水 |
| 200ms | 可用但偏慢 |
| **100ms** | **平衡点** |

## 阶段 3：死亡检测系统演进

### 3.1 初始状态：单帧 HSV 检测

```python
# _count_red_pixels 使用 PIL HSV
# HEART_DEATH_FRAMES = 3（3帧连续丢失即死亡）
```

**问题**：
1. PIL H 通道 0-255 ≠ OpenCV 0-180 → H 阈值偏移 → 红色检测不准
2. 单帧抖动重置计数器 → 频繁误判
3. 无法区分死亡 vs 广告弹窗

### 3.2 BUG 发现与修复：PIL HSV 范围不匹配

**排查**：红心明明在画面中，但 `_count_red_pixels` 返回值极低。

**根因**：config 阈值 H_MAX1=10 是按 OpenCV 范围（10/180×360°=20°），但 PIL 的 H 是 0-255 范围（10/255×360°=14°）。红色检测范围缩窄了 40%。

**修复**：改用 `cv2.cvtColor(rgb, cv2.COLOR_RGB2HSV)`。

### 3.3 改进 1：多帧投票 + 平滑衰减

```python
# 训练: 3帧多数投票（≥2/3无红心→计丢失）
# 采集: 单帧快速
# 有红心时计数器-1而非清零（平滑衰减）
```
防止渲染闪烁误判。

### 3.4 改进 2：二次验证区分广告

**问题**：安全帽弹窗、随机广告也会导致红心消失。

**方案**：红心丢失后 → 点「取消/放弃」(134,1850) → 等 1s → 查 5 帧红心：
- 红心回来 → 广告关了 → 继续
- 红心没回 → 真死亡 → 重启

**关键发现**：死亡界面的「放弃」和广告弹窗的「取消」恰好同位置(134,1850)。

### 3.5 改进 3：训练 step() 重构

**问题**：`step()` 先发动作再检测死亡 → "Step 36 显示 swipe_up, reward=-10"（人死了还发滑动）。

**修复**：重构为先截屏 → 检测死亡（含 _verify_death）→ 死了直接 return → 活着才执行动作。

### 3.6 时序优化

**问题**：红心一丢就点按钮 → UI 还没加载出来 → tap 打在空气上。

**修复**：在 _verify_death 中先 `time.sleep(0.8)` 等 UI 加载。

## 阶段 4：数据采集系统演进

### v1：批量读键 + 批次截屏
- 清空键盘缓冲区 → 处理所有键 → 截一次屏
- **问题**：send_action 期间新按键丢失

### v2：加冷却间隔 MIN_ACTION_INTERVAL=0.35s
- 冷却期按键丢弃
- **问题**：自动 no_op 也更新冷却计时器 → 用户按键被误丢弃

### v3：冷却改为 sleep 等待
- 不丢弃按键，sleep 等待冷却结束
- **问题**：sleep 期间读不到新键

### v4：异步架构（失败）
- 主线程读键发滑动，后台线程截屏+录 no_op
- **致命缺陷**：`_verify_death()` 在后台线程阻塞 2.2s
- **证据**：采集 241 条数据，no_op 仅 12.9%（正常 40-60%）

### v5：极简同步（当前）
- 单线程：读键→发滑动→observe→循环
- 无冷却、无线程、不存截图
- _verify_death 阻塞主线程（合理——死了就该等）

## 阶段 6：数据保存策略改进

### 旧方案问题
- 固定路径 data/states.npy + data/actions.npy
- 不加 --resume 直接覆盖
- 用户多次采集间数据丢失

### 新方案
- 每次采集独立目录 data/session_YYYYMMDD_HHMMSS/
- 永不覆盖
- 自动生成 meta.txt（动作分布统计）
- --resume 续接最近会话

## 关键技术决策总结

| 决策 | 方案 | 替代方案 | 选择理由 |
|------|------|---------|---------|
| 截屏 | minicap 15fps socket | ADB screencap / DXGI | 原生分辨率+低延迟 |
| 触控 | Popen exec-out | subprocess.run / 持久化shell | 简单可靠，~5ms |
| 死亡检测 | HSV 3帧投票+二次验证 | 单帧检测 / 冻结检测 | 鲁棒性+区分广告 |
| 数据采集 | 单线程同步 | 异步架构 | 简单性，避免竞态 |
| 数据保存 | session 目录 | 固定路径 | 防覆盖 |
| 训练管理 | runs/train/expN | 固定路径 | 实验可追溯 |
