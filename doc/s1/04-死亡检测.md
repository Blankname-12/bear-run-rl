# 04 — 死亡检测与游戏重启

## 问题定义

RL 训练中，智能体需要知道何时"死亡"以触发环境重置。人类玩家通过游戏画面判断，AI 需要自动检测。

熊大快跑在左上角有心形生命指示器（红心）。角色存活时红心存在；角色死亡或弹出广告/安全帽弹窗时红心消失。因此"红心是否存在"是判断游戏状态的核心信号。

**但仅有红心检测不够**——因为广告和安全帽弹窗也会让红心消失（弹窗盖住了画面）。

---

## 红心 HSV 检测原理

### 区域定义

屏幕左上角 (0, 0) → (120, 50)，原生 1080×1920 坐标。这是一个很小的区域（120×50 = 6000 像素），足以包含红心图标。

```python
HEART_REGION_LEFT = 0
HEART_REGION_TOP = 0
HEART_REGION_RIGHT = 120
HEART_REGION_BOTTOM = 50
```

### 为什么用 HSV 而非 RGB

RGB 对光照变化敏感（游戏画面亮暗变化会改变红色的 RGB 值）。HSV 将色彩（Hue）、饱和度（Saturation）、亮度（Value）分离：
- 红色在 HSV 中对应特定色相角（0° 和 360°），与亮度/饱和度无关
- 饱和度和亮度可以单独设阈值，过滤非纯红色像素

### OpenCV HSV 检测代码

```python
def _count_red_pixels(rgb_patch):
    import cv2
    hsv = cv2.cvtColor(rgb_patch, cv2.COLOR_RGB2HSV)
    h = hsv[..., 0]  # H: 0-180 (OpenCV范围)
    s = hsv[..., 1]  # S: 0-255
    v = hsv[..., 2]  # V: 0-255

    # 红色跨越 HSV 0°，需要两个区间
    mask1 = (h >= 0)  & (h <= 10)  & (s >= 80) & (v >= 80)  # 0°附近
    mask2 = (h >= 170) & (h <= 180) & (s >= 80) & (v >= 80)  # 360°附近

    return int(np.sum(mask1) + np.sum(mask2))
```

**为什么红色需要两个区间**：
HSV 色相环是圆形的，红色恰好在 0°/360° 的分界线上。OpenCV 中 0-180 对应实际 0-360°，红色分布在 0-10（靠近 0°）和 170-180（靠近 360°）。

### 阈值选择依据

| 参数 | 当前值 | 选择理由 |
|------|--------|---------|
| H 范围 | [0,10] ∪ [170,180] | 只选纯红色，排除橙色(H≈15-30)和粉色(H≈150-165) |
| S_MIN | 80 | 排除粉色(S低)、灰色(S≈0)，确保是鲜艳的红色 |
| V_MIN | 80 | 排除暗红色(V低)，红心生命值即使闪烁也足够亮 |
| HEART_PIXEL_RATIO | 0.005 | 即 6000 像素中 ≥ 30 个红像素。一个红心图标通常有 200-500 个红像素 |

### S_MIN/V_MIN 演变（补偿 DXGI → 恢复严格）

| 阶段 | S_MIN | V_MIN | 原因 |
|------|-------|-------|------|
| 早期 (ADB) | 60 | 80 | 初始经验值 |
| DXGI 截图时 | **40** | **50** | DXGI 截图色彩偏淡、饱和度低，必须放宽 |
| minicap 截图时 | **80** | **80** | minicap 原生截屏色彩准确，恢复严格值 |

值的注意：当初切换到 minicap 后忘记把 S_MIN 从 40 改回 80。结果几乎所有暗色像素（灰色UI边框、阴影）都满足 S≥40 的条件，导致 `_count_red_pixels` 把大量非红色像素也计入了。红心即使不存在，粉色背景的 UI 元素也能提供足够多的"红像素"，导致**死亡检测失效**（检测不到死亡）。

### 关键 BUG：PIL vs OpenCV HSV 范围差异

这个问题非常隐蔽，调试了很久才发现。

**PIL HSV 转换**：
```python
hsv = np.array(Image.fromarray(rgb).convert("HSV"))
# H: 0-255 (对应 0-360°)
# OpenCV 的 H=10 → 实际 20°
# PIL 的 H=10 → 实际 10×360/255 ≈ 14°
```

**OpenCV HSV 转换**：
```python
hsv = cv2.cvtColor(rgb, cv2.COLOR_RGB2HSV)
# H: 0-180 (对应 0-360°)
# H=10 → 实际 20°
```

config.py 中 `HEART_H_MAX1 = 10` 是按 OpenCV 范围设计的（对应 20°）。早期代码使用 PIL 做 HSV 转换，H=10 在 PIL 中仅对应 14°，红色检测范围缩小了 40%。这导致红心即使正常显示，检测到的红色像素数量也偏少，频繁触发假死亡。

**修复**：将 `_count_red_pixels` 中的 PIL HSV 转换改为 `cv2.cvtColor(rgb, cv2.COLOR_RGB2HSV)`，使 H 阈值与设计值一致。

## 两阶段死亡判定

### 阶段一：心跳丢失检测 `_heart_lost(raw, fast)`

```python
def _heart_lost(self, raw_frame, fast=False):
    sx, sy = _capture_scale or (1.0, 1.0)
    region_area = (HEART_REGION_BOTTOM - HEART_REGION_TOP) * sy * \
                  (HEART_REGION_RIGHT - HEART_REGION_LEFT) * sx

    def _frame_no_heart(frame):
        return _detect_heart_pixels(frame) / region_area < HEART_PIXEL_RATIO

    if fast:
        lost = _frame_no_heart(raw_frame)                # 采集：单帧
    else:
        lost_count = int(_frame_no_heart(raw_frame))     # 训练：3帧投票
        for _ in range(2):
            time.sleep(0.03)
            lost_count += int(_frame_no_heart(get_screen_numpy()))
        lost = lost_count >= 2  # ≥2/3 帧无红心 → 确认丢失

    if lost:
        self._heart_loss_frames += 1
    else:
        self._heart_loss_frames = max(0, self._heart_loss_frames - 1)

    if self._heart_loss_frames >= HEART_DEATH_FRAMES:
        self._heart_loss_frames = 0
        return True
    return False
```

**3 帧多数投票（训练模式）**：
- 连取 3 帧（间隔 30ms），每帧做 HSV 检测
- ≥2/3 帧无红心 → 才认为本轮"红心丢失"
- 防止单个渲染抖动（如粒子特效遮挡红心 1 帧）触发误判

**平滑衰减 `max(0, counter - 1)`**：
- 红心恢复时计数器 -1 而非直接清零
- 如果红心短暂出现 1 帧（UI 刷新闪烁），计数器只从 9 降到 8，不会从 9 跳到 0 重新开始
- 需要连续恢复 HEART_DEATH_FRAMES 帧才能把计数器降回 0

**单帧快速（采集模式）**：
- 采集模式人类在看着画面，不需要投票
- 直接用当前帧判断，如果真死了人类也能感觉到并手动干预

### 阶段二：二次验证 `_verify_death()`

红心连续丢失达到阈值后，不立即判定死亡。因为存在两种可能：
1. **真死亡**：角色撞了，游戏弹出死亡界面
2. **广告/安全帽弹窗**：游戏弹出弹窗盖住了红心，但角色并未死亡

```python
def _verify_death(self):
    """红心丢失后的二次检验"""
    time.sleep(0.8)  # 等死亡UI/广告弹窗完全加载
    print("[Env] 红心丢失，点「取消」测试...")

    # 点 (134,1850) — 同一个位置：
    #   - 如果是死亡界面 → 点的是「放弃」
    #   - 如果是广告弹窗 → 点的是「取消」
    self._tap(GIVE_UP_BUTTON_X, GIVE_UP_BUTTON_Y)
    time.sleep(1.0)  # 等点击生效和UI过渡动画

    # 连续查 5 帧红心
    for _ in range(5):
        test_raw = get_screen_numpy()
        heart_px = _detect_heart_pixels(test_raw)
        if heart_px / region_area >= HEART_PIXEL_RATIO:
            print("[Env] 红心恢复 → 是广告弹窗，继续游戏")
            self._heart_loss_frames = 0
            return False  # 广告已关，继续游戏
        time.sleep(0.08)

    print("[Env] 红心未恢复 → 确认死亡")
    self._heart_loss_frames = 0
    return True  # 确认死亡
```

**为什么点 (134, 1850) 能区分**：

这是本系统设计中最幸运的巧合。经实测发现：
- 死亡界面：「放弃」按钮位置 = (134, 1850)
- 安全帽/广告弹窗：「取消/关闭」按钮也在 (134, 1850)

同一个 tap，在两种场景下分别触发不同的结果。配合 click 后红心回归检测，完美区分两者。

**时序设计**：
```
红心丢失检测
  ↓ 等 0.8s — UI 加载时间（死亡动画/弹窗显现）
  ↓ 点 (134, 1850)
  ↓ 等 1.0s — 点击生效 + UI 过渡（弹窗关闭动画/死亡界面切换到计分界面）
  ↓ 查 5 帧红心 — 每帧间隔 80ms，共 0.4s
  ↓
判定结果
```
总计约 2.2 秒的验证过程。这在训练中是可以接受的（只在死亡时触发）。

### 训练模式 step() 中的集成

```python
def step(self, action_idx):
    raw = get_screen_numpy()     # 1. 先截图

    if not warmup:
        heart_lost = self._heart_lost(raw)   # 2. 检测心跳（3帧投票）
        if heart_lost:
            if self._verify_death():          # 3. 二次验证（~2.2s）
                return state, REWARD_DEATH, True  # 4. 真死 → 返回

        # 5. 活着 → 执行动作
        action_swipe(action_idx)
        ...
```

关键设计：**动作执行在死亡检测之后**。早期版本是先发动作再检测，导致"Step 36 显示 swipe_up, reward=-10"——角色已死还在发滑动指令。

### 采集模式 observe() 中的集成

```python
def observe(self, save_ss=True):
    raw = get_screen_numpy()
    ...
    if not warmup:
        if self._heart_lost(raw, fast=True):  # 单帧快速
            done = self._verify_death()        # 阻塞验证
    return self.get_state(), raw, done
```

采集模式用 fast=True（单帧），减少每步开销。活着时每秒可完成更多操作。

## 重启流程

```python
def restart_game(self):
    """真死亡后的重启流程（_verify_death 已经点了「放弃」）"""
    time.sleep(1.5)  # 等放弃界面过渡

    # 点「重新开始」
    self._tap(RESTART_BUTTON_X, RESTART_BUTTON_Y)  # (544, 1800)
    time.sleep(0.4 + 1.0)  # 按钮响应 + UI过渡

    # 点「开始跑酷」
    self._tap(START_BUTTON_X, START_BUTTON_Y)  # (698, 1491)
    time.sleep(0.4 + 1.5)

    # reset() → 清状态 → 设 warmup → 新局开始
```

总耗时约 4.3 秒。加上 `_verify_death` 的 2.2 秒，从死亡到新局开始约 6.5 秒。

### 按钮配置

| 按钮 | 坐标 | 作用 |
|------|------|------|
| 放弃/取消 | (134, 1850) | 死亡界面点放弃 / 弹窗点取消 |
| 重新开始 | (544, 1800) | 计分界面的重开按钮 |
| 开始跑酷 | (698, 1491) | 角色选择后的确认按钮 |

### 开局保护（Warmup）

```python
GAME_START_WARMUP_SECONDS = 8.0  # 开局前8秒不检测死亡
```

新局开始后，游戏有加载动画（角色出场、UI 展示）。期间红心可能未渲染或被动画遮挡。8 秒保护期确保不会误判死亡。

在 step() 中：
```python
warmup = time.time() < self._warmup_deadline
if warmup:
    return next_state, 0.0, False  # 不检测死亡，不执行动作，返回 0 奖励
```

在 observe() 中：
```python
warmup = time.time() < self._warmup_deadline
done = False if warmup else self._verify_death()
```
