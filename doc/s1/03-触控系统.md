# 03 — 触控系统

## 问题定义

RL 智能体通过 ADB 向模拟器发送触控指令（滑动/点击）来控制游戏。触控延迟直接影响：
1. 训练效率：延迟高 → 每步耗时长 → 训练慢
2. 数据采集质量：延迟高 → 人类操作体验差 → 采集数据质量下降
3. 游戏响应：指令必须及时到达设备，否则角色已撞上障碍物

**核心需求**：Python 端指令发出后，设备端能在 50ms 内开始执行。

---

## 方案一：subprocess.run + adb shell（初始 · 已废弃）

### 实现
```python
subprocess.run(
    ["adb", "-s", serial, "shell", "input", "swipe",
     str(x1), str(y1), str(x2), str(y2), str(duration_ms)],
    capture_output=True, check=True, timeout=5
)
```

### 延迟分解

在 Windows 上实测：
```
subprocess.run 总耗时: 182ms
  进程创建 (Windows CreateProcess):  ~50ms
  ADB client → ADB server 通信:      ~20ms
  ADB server → 设备 PTY 分配:        ~30ms
  设备端 shell 解析命令:              ~30ms
  input swipe 执行:                  SWIPE_DURATION_MS (50ms)
  ADB 协议开销:                      ~2ms
```

182ms 对实时游戏操控来说太慢了。每次滑动都要等将近 200ms。

### 尝试优化：exec-out 替代 shell

```python
subprocess.run(
    ["adb", "-s", serial, "exec-out", "input", "swipe", ...],
    capture_output=True, timeout=5
)
```

`exec-out` 不分配 PTY，直接管道通信：
```
subprocess.run + exec-out: 55ms
  进程创建:                   ~50ms
  ADB 通信:                   ~3ms
  设备端执行:                 SWIPE_DURATION_MS
```

比 shell 快 3.3 倍（55ms vs 182ms），但仍然阻塞等待。如果用 `subprocess.run`，每步都要等 55ms。

## 方案二：持久化 ADB Shell（尝试 · 失败）

### 设计

维护一个长期 `adb shell` 进程，通过 stdin 管道持续发送命令，避免每次 spawn 进程。

```python
proc = subprocess.Popen(
    ["adb", "-s", serial, "shell"],
    stdin=subprocess.PIPE,
    stdout=subprocess.DEVNULL,
    stderr=subprocess.DEVNULL,
)
# 发命令
proc.stdin.write(b"input swipe 540 1344 540 576 50\n")
proc.stdin.flush()
```

### 失败原因

在 Windows 上，`adb shell` 检测到 stdin 不是 TTY（是管道）后立即退出。实测 `proc.poll()` 在 0.3s 后返回 exit code 1。

尝试了替代方案：
1. `adb shell "while read cmd; do eval \$cmd; done"` — 进程立即退出
2. `adb shell -T`（强制 PTY）— 进程存活但无法通过管道写入
3. `adb shell cat` — 不执行命令，只能 echo
4. `adb shell toybox sh -c '...'` — toybox 无此问题但仍退出

结论：ADB shell 在 Windows 上的 TTY 检测机制导致无法使用管道持久化方案。**此方向放弃。**

## 方案三：subprocess.Popen + adb exec-out（最终方案）

### 设计

用 `subprocess.Popen`（非阻塞）配合 `exec-out`（绕过 PTY），实现 fire-and-forget 模式：

```python
subprocess.Popen(
    ["adb", "-s", serial, "exec-out", "input", "swipe",
     str(x1), str(y1), str(x2), str(y2), str(duration_ms)],
    stdout=subprocess.DEVNULL,
    stderr=subprocess.DEVNULL,
)
```

Python 端耗时 ~5ms（仅 Popen 进程 spawn），设备端约 55ms 后开始执行。Python 不等待执行结果，立即返回处理下一帧。

### FastTouch 类完整实现（fast_io.py）

```python
class FastTouch:
    """使用 adb exec-out 发送触控指令（绕过 PTY，比 shell 快 3x）。
    Popen fire-and-forget: 指令在 ~5ms 内发出，设备端 ~55ms 内执行。"""

    def __init__(self, device_serial):
        self.serial = device_serial

    def start(self):
        pass  # 无需启动持久化进程

    def _exec(self, *args):
        """fire-and-forget"""
        subprocess.Popen(
            ["adb", "-s", self.serial, "exec-out"] + list(args),
            stdout=subprocess.DEVNULL,
            stderr=subprocess.DEVNULL,
        )

    def _exec_sync(self, *args):
        """同步执行，等待 ADB 确认。用于重启流程的关键按钮点击。"""
        subprocess.run(
            ["adb", "-s", self.serial, "exec-out"] + list(args),
            capture_output=True, timeout=5,
        )

    def swipe(self, x1, y1, x2, y2, duration_ms):
        """异步滑动（游戏操作），~5ms 返回"""
        self._exec("input", "swipe",
                   str(x1), str(y1), str(x2), str(y2), str(duration_ms))

    def tap(self, x, y):
        """异步点击（游戏操作），~5ms 返回"""
        self._exec("input", "tap", str(x), str(y))

    def tap_sync(self, x, y):
        """同步点击（重启按钮），等待设备确认执行完成"""
        self._exec_sync("input", "tap", str(x), str(y))

    def close(self):
        pass
```

### 同步 vs 异步的区别

| 方法 | 实现 | 耗时 | 使用场景 |
|------|------|------|---------|
| `swipe()` | Popen fire-and-forget | ~5ms | 游戏滑动操作（跳/滑/左右） |
| `tap()` | Popen fire-and-forget | ~5ms | 游戏点击 |
| `tap_sync()` | subprocess.run 等待 | ~55ms | 重启流程按钮（放弃/重开/开跑） |

**为什么重启按钮需要同步**：按钮点击后游戏 UI 会切换，必须确保上一次点击成功执行后才能发下一次点击。如果异步发，三个点击可能同时到达，导致游戏 UI 状态混乱。

## SWIPE_DURATION_MS 参数调优

Android `input swipe` 的最后一个参数是滑动持续时间（毫秒）。这个值决定了设备端触控事件的采样密度和手势识别结果。

### 调参记录

| 值 | 现象 | 分析 |
|------|------|------|
| **50ms** | 滑动经常被识别为点击 | Android 手势识别器在极短滑动距离+极短时间下会判定为 tap |
| **80ms** | 仍有部分识别为点击 | 临界值，不够稳定 |
| **300ms** | 明确识别为滑动 | 但滑动太慢，延迟感知明显（玩家感受"拖泥带水"） |
| **200ms** | 识别为滑动，延迟可接受 | 用户尝试后觉得还行 |
| **100ms** | 识别为滑动，延迟低 | **当前值** |

### Android 手势识别机制

Android 的 `input swipe` 命令会在指定的 duration 内生成一系列触控事件。手势识别器根据以下条件判定：

```
判定为 swipe 的条件：
1. 触控点移动距离 > touchSlop（通常 8dp ≈ 24px）
2. 持续时间 ≥ 阈值（约 80-100ms）
3. 触控点速度曲线一致

判定为 tap 的条件：
1. 触控点移动距离 < touchSlop
2. 持续时间短
3. 或者移动距离虽够但时间太短（触控点"跳跃"而非"滑动"）
```

在 1080×1920 屏幕上，我们的滑动距离约 300-768 像素。100ms 足够完成这一距离而不被误判为 tap。

### 对训练的影响

`SWIPE_DURATION_MS` 影响训练中每步的实际时长：
- `step()` 中 `action_swipe()` 发送 Popen 后立即返回（~5ms），不等待 100ms 滑动完成
- 但 `observe()` 的 `_heart_lost()` 在训练模式下额外捕获 2 帧（每帧 ~20ms）

实际的训练步长约 30-50ms，主要由截屏决定而非触控。

## 滑动坐标设计

基于 1080×1920 竖屏，中心点为 (540, 960)。

| 动作 | 起点 | 终点 | 滑动距离 | 含义 |
|------|------|------|---------|------|
| swipe_up (跳) | (540, 1344) | (540, 576) | 768px | 屏幕 70%→30%，模拟手指从下向上划 |
| swipe_down (滑) | (540, 576) | (540, 1344) | 768px | 屏幕 30%→70%，模拟手指从上向下划 |
| swipe_left (左) | (594, 960) | (486, 960) | 108px | 屏幕 55%→45%，模拟手指从右向左划 |
| swipe_right (右) | (486, 960) | (594, 960) | 108px | 屏幕 45%→55%，模拟手指从左向右划 |

```python
SCREEN_WIDTH = 1080
SCREEN_HEIGHT = 1920

# swipe_up 坐标计算
cx = SCREEN_WIDTH // 2    # 540
y1 = int(SCREEN_HEIGHT * 0.7)  # 1344
y2 = int(SCREEN_HEIGHT * 0.3)  # 576
# swipe: (540, 1344) → (540, 576), duration=100ms
```

左右滑动的距离（108px）比上下滑动（768px）短很多，因为赛道切换只需要一个短促的横滑，而跳跃/滑铲需要明显的手势来触发游戏的动作识别。

## 动作空间设计

| 索引 | 名称 | 键盘 | 含义 | Android 指令 |
|------|------|------|------|-------------|
| 0 | swipe_up | W / ↑ | 跳跃 | input swipe 540 1344 540 576 100 |
| 1 | swipe_down | S / ↓ | 滑铲 | input swipe 540 576 540 1344 100 |
| 2 | swipe_left | A / ← | 左移 | input swipe 594 960 486 960 100 |
| 3 | swipe_right | D / → | 右移 | input swipe 486 960 594 960 100 |
| 4 | no_op | (不按键) | 不动 | (无 ADB 指令) |

### no_op 的重要性

早期版本只有 4 个动作（无 no_op），意味着**智能体每一帧都必须选择一个滑动方向**。这导致：
1. 模型永远学不会"直道上不用操作"
2. 训练数据全部是主动操作，模型不知道"不动"也是合理策略
3. 游戏角色不停抖动（一直发滑动指令）

增加 no_op（action=4）后：
- 模型可以在无障碍物时选择 no_op
- 人类采集数据中 no_op 自然占到 40-60%（大部分时间在直跑）
- 游戏角色行为更自然

在 `game_env.py` 中 no_op 的处理：
```python
# send_action 中
elif action_idx == 4:      # no-op — 不动
    return                 # 直接返回，不发送任何 ADB 指令

# action_swipe 中（训练用）
elif action_idx == 4:
    return
```

no_op 不发送任何 ADB 指令，也不改变赛道位置追踪。

## 训练 vs 采集的触控差异

### 采集模式（collect_data.py）

```python
env.send_action(action_idx)  # Popen fire-and-forget, ~5ms
state, _, done = env.observe(save_ss=False)  # 截屏+检测, ~20ms
```

用户按键 → 立刻发滑动 → 观察下一帧 → 可以立刻按下一个键。无冷却、无延迟。

### 训练模式（train.py step）

```python
def step(self, action_idx):
    raw = get_screen_numpy()            # 先截屏
    ...
    heart_lost = self._heart_lost(raw)  # 检测心跳
    if heart_lost:
        if self._verify_death():        # 确认死亡 (~2s)
            return ..., -10.0, True     # 直接返回，不发动作

    action_swipe(action_idx)            # 活着，执行动作
    ...
```

训练模式的 `step()` **先检测死亡再执行动作**——如果角色已死，不发送滑动指令。这是为了避免"人死了还在发 swipe"的问题。

## 性能汇总

| 方案 | Python端延迟 | 设备端延迟 | 状态 |
|------|------------|-----------|------|
| subprocess.run + shell | 182ms | 50ms | 废弃 |
| subprocess.run + exec-out | 55ms | 50ms | 废弃 |
| 持久化 ADB shell | N/A | N/A | 失败 |
| **Popen + exec-out** | **~5ms** | **~55ms** | **当前** |

Python 端 ~5ms 的开销对训练吞吐影响微乎其微。每步 30-50ms 的总时间主要由 minicap 截屏决定。
