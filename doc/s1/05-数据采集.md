# 05 — 数据采集系统

## 目标

通过人类玩家手动操控游戏，采集 (state, action) 数据对，用于行为克隆预训练。采集质量直接影响模型学习效果。

## 核心挑战

1. **实时性**：玩家按下 W/A/S/D 后必须立刻响应，延迟会破坏操作体验
2. **数据对齐**：记录的 state 必须是"玩家做决定时看到的画面"，而非"操作执行后的画面"
3. **no-op 比例**：游戏中大部分时间是直线跑（no_op），如果 no_op 采集不足，模型会认为"每帧都要操作"
4. **可靠性**：采集过程中不能卡住、不能无声崩溃、不能让玩家白玩

## 五版迭代

### v1：批量读键 + 批次截屏（废弃）

```python
# 清空键盘缓冲区
pending_keys = []
while msvcrt.kbhit():
    pending_keys.append(read_keyboard())

# 逐键处理
for key in pending_keys:
    send_action(key_to_action(key))
    # 在发送期间继续吸新按键
    while msvcrt.kbhit():
        pending_keys.append(read_keyboard())

# 所有键处理完后截一次屏
state, raw, done = env.observe()
```

**问题**：处理每个键的 send_action 需要约 5ms（Popen spawn），期间即使有 `while msvcrt.kbhit()` 吸新按键，仍有短暂的丢失窗口。更关键的是截屏只在批次末尾做一次，中间键的状态没有更新。

### v2：加冷却间隔（废弃）

```python
MIN_ACTION_INTERVAL = 0.35  # 两次操作最小间隔
if time.time() - _last_action_time < MIN_ACTION_INTERVAL:
    continue  # 冷却期丢弃按键
```

**问题**：自动 no-op 录制也更新 `_last_action_time`，导致：
1. 玩家直线跑 2 秒（自动录了 5 个 no-op，`_last_action_time` 一直在更新）
2. 玩家想按 W → `elapsed = 0.1s < 0.35s` → 按键被丢弃
3. 玩家感受"按键没反应"，多按几次，可能的都被丢弃

### v3：冷却改 sleep 等待（废弃）

将 `continue`（丢弃）改为 `time.sleep()`（等待）。但 sleep 期间读不到按键，且自动 no-op 仍干扰冷却计时。

### v4：异步架构（废弃 · 致命缺陷）

```python
# 主线程：只管键盘和发滑动
while True:
    key = read_keyboard()
    if action in (0,1,2,3):
        env.send_action(action)  # fire-and-forget
        states.append(state.copy())  # 用后台线程更新的 state
        actions.append(action)

# 后台线程：持续截屏+更新state+检测死亡+录no-op
def _capture_loop():
    while running:
        raw = get_screen_numpy()
        state = env.get_state()
        heart_gone = env._heart_lost(raw, fast=True)
        if heart_gone:
            if env._verify_death():  # ⚠️ 阻塞 2.2 秒！
                _done_flag[0] = True
        # 录 no-op
        if now - last_noop >= 0.4:
            states.append(state); actions.append(4)
```

**致命问题**：`_verify_death()` 在红心丢失时阻塞后台线程 **2.2 秒**。期间：
1. 不截屏 → state 不更新
2. 不录 no-op → no_op 大量丢失
3. 主线程继续按键 → 动作照录，no_op 全丢

**数据佐证**：采集 241 条数据，no_op 仅 31 条（12.9%）。正常应为 40-60%。

### v5：极简同步（当前版本）

**设计原则**：
- 无线程 → 无竞态条件
- 不存截图 → 砍 IO 延迟
- `_verify_death()` 阻塞主线程 → 死了就该等
- 无冷却 → 手动操作不需要限速
- 异常打印 traceback → 崩溃可见

**核心循环**：

```python
_last_noop_time = 0
state = env.reset()

while True:
    key = read_keyboard()

    # ── 没按键：定期录 no-op ──
    if key is None:
        now = time.time()
        if now - _last_noop_time >= NOOP_OBSERVE_INTERVAL:  # 0.4s
            next_state, raw, done = env.observe(save_ss=False)
            if done:
                env.restart_game()
                state = env.reset()
            else:
                state = next_state
            _last_noop_time = now
            states_list.append(state.copy())
            actions_list.append(4)  # no_op
        else:
            time.sleep(0.01)
        continue

    # ── 有按键 ──
    action = key_to_action(key)
    if action == -1: break       # Q
    if action == -2: restart     # R
    if action in (0,1,2,3):
        # 记录(当前状态, 动作)
        states_list.append(state.copy())
        actions_list.append(action)
        # 发滑动（Popen, ~5ms）
        env.send_action(action)
        # 观察下一帧
        next_state, raw, done = env.observe(save_ss=False)
        if done:
            env.restart_game()
            state = env.reset()
        else:
            state = next_state
        _last_noop_time = time.time()
```

**关键时序**：
1. 读键（~0ms）
2. 记录 (state_before_action, action)（~0ms）
3. `env.send_action(action)` Popen fire-and-forget（~5ms）
4. `env.observe(save_ss=False)` minicap drain + 截屏 + 检测（~30ms）
5. 回到步骤 1 读下一个键

单步总耗时 ~35ms，支持约 30 次操作/秒。人类操作频率通常 2-5 次/秒，完全够用。

## 数据格式

### states.npy
```
形状: (N, 15, 128, 128)
类型: float32
范围: [0.0, 1.0]

通道分配:
  0-2:   第1帧 RGB (128×128)  ← 最近帧
  3-5:   第2帧 RGB
  6-8:   第3帧 RGB
  9-11:  第4帧 RGB            ← 最远帧（0.12-0.2秒前）
  12:    赛道左 one-hot (1.0 or 0.0)
  13:    赛道中 one-hot
  14:    赛道右 one-hot
```

**为什么堆叠 4 帧**：单帧无法表示运动信息。障碍物从远处逼近，4 帧的时间跨度（~200ms）足以让 CNN 学习障碍物的移动方向和速度。

**赛道 one-hot**：游戏有三条赛道（左/中/右），agent 需要知道自己在哪条赛道上，以决定是否需要切换。one-hot 编码直接附加到帧堆叠后面作为额外通道。

### actions.npy
```
形状: (N,)
类型: int64
值: 0=跳, 1=滑, 2=左, 3=右, 4=no_op
```

### meta.txt
```
采集时间: 2026-05-15 21:55:39
总记录数: 241
动作分布:
  swipe_up    :    57 (23.7%)
  swipe_down  :    31 (12.9%)
  swipe_left  :    63 (26.1%)
  swipe_right :    59 (24.5%)
  no_op       :    31 (12.9%)
```

meta.txt 在保存时自动生成，方便查看数据分布是否合理。

## 图像预处理管线

```
1080×1920 uint8 RGB (minicap)
  ↓ _crop_game_area()
约 960×1500 游戏区域（去UI、去上下边框）
  ↓ cv2.resize(128, 128, INTER_LINEAR)
128×128×3
  ↓ .astype(np.float32) / 255.0
[0, 1] float32
  ↓ .transpose(2, 0, 1)
(3, 128, 128) CHW 格式（PyTorch 标准）
```

**为什么用 OpenCV 而非 PIL**：`cv2.resize` 比 `PIL.Image.resize` 快约 5 倍（使用 SIMD 优化的 C++ 实现）。

```python
def preprocess_frame(frame):
    cropped = _crop_game_area(frame)
    import cv2
    resized = cv2.resize(cropped, (128, 128), interpolation=cv2.INTER_LINEAR)
    return resized.astype(np.float32).transpose(2, 0, 1) / 255.0
```

## 数据保存策略

- 每次采集创建独立目录 `data/session_YYYYMMDD_HHMMSS/`
- **永不覆盖**（vs 旧版直接覆盖 data/states.npy）
- `--resume` 参数续接到最近会话
- 退出时自动生成 meta.txt
