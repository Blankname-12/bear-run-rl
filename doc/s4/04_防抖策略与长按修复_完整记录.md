# S4-04 防抖策略与长按修复 —— 完整记录

## 第一部分：play_ai.py 防抖策略演进

### 1.1 v1：激进防抖

```python
def should_act(self, action, confidence):
    if action == 5:
        return confidence >= 0.90          # 长按门槛
    if confidence < 0.30:                  # 低置信度
        return False
    if now - self._last_action_time < 0.25: # 时间间隔
        return False
    if action == self._last_action and action in (0,1,2,3): # 同动作阻断
        return False
    return True
```

**问题**：三个过滤叠加（置信度 + 时间间隔 + 同动作阻断），AI 基本不动。用户痛骂："能不能把这个防抖去掉？"

### 1.2 v2：全去掉

用户最初说"需要保留低置信度过滤，只需要删除防抖"：

```python
def should_act(self, action, confidence):
    if action == 5:
        return confidence >= 0.90
    if confidence < 0.30:
        return False
    return True  # 无时间间隔、无同动作阻断
```

### 1.3 v3：智能防抖（当前方案）

用户最终要求："如果两次连续操作都是相同的，那就第二个舍弃，如果三个连续的，第三个及以后不舍弃"

**核心逻辑**：
```
连续第 1 次动作 X → 放行
连续第 2 次动作 X → 跳过（防双击）
连续第 3 次动作 X → 放行（允许持续按住不放）
动作变为 Y       → 计数重置，立即放行
```

**完整实现**：

```python
class AIPlayer:
    def __init__(self, ...):
        self._same_count = 0  # 新增：连续相同动作计数器

    def reset(self):
        self._same_count = 0  # 重置时清零

    def should_act(self, action, confidence):
        # 长按仅在置信度 >= 90% 时执行（误触代价高，浪费道具）
        if action == 5:
            return confidence >= 0.90

        # 低置信度（< 30%）跳过
        if confidence < 0.30:
            return False

        # 智能防抖
        if action == self._last_action:
            self._same_count += 1
            if self._same_count == 2:
                return False  # 连续第 2 次 → 跳过！
        else:
            self._same_count = 1  # 换了动作 → 重置计数

        return True
```

**执行时更新**：

```python
# 主循环中
if ai.should_act(action, confidence):
    ai.execute(action, confidence)     # 执行 + 更新 _last_action
    action_history.append(action)
elif action is not None:
    action_history.append(4)           # 记录为 noop
    if action != 4 and action != last_print_action:
        print(f"跳过:置信度低 {confidence:.1%}")
last_print_action = action
```

**其他过滤（execute 中边界拦截）**：
```python
def execute(self, action, confidence):
    if action == 2 and self._current_lane == 1:   # 已在最左，不发左移
        return
    if action == 3 and self._current_lane == 3:   # 已在最右，不发右移
        return
    send_action(action)
    self._last_action = action
```

## 第二部分：长按修复

### 2.1 问题现象

用户运行 `play_ai.py` 时发现："长按 1 秒被识别成快速点击，根本就没有长按"。即游戏把长时间触摸当作连续快速点击（点两下）。

### 2.2 根因：双层循环 + Popen 不等 × 9 个重叠命令

**bug 链路追踪**：

**第一层（game_env.py send_action）**：
```python
# game_env.py - send_action(5)
elif action_idx == 5:      # 长按
    for _ in range(config.LONG_PRESS_REPEAT):  # LONG_PRESS_REPEAT = 3
        ft.long_press(cx, cy, config.LONG_PRESS_DURATION_MS)  # 1000ms
        time.sleep(0.15)
```
循环 3 次，每次间隔 0.15 秒。

**第二层（fast_io.py FastTouch.long_press）**：
```python
# fast_io.py - 旧版 long_press
def long_press(self, x, y, duration_ms):
    for _ in range(config.LONG_PRESS_REPEAT):  # LONG_PRESS_REPEAT = 3
        subprocess.Popen(                       # ← Popen！不等待！
            ["adb", "shell", "input", "swipe",
             str(x), str(y), str(x), str(y), str(duration_ms)],  # 1000ms
            stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL,
        )
        time.sleep(0.15)
```
循环 3 次，每次 fire-and-forget（Popen），不等命令完成。

**实际时间线**：
```
t=0.00s:  game_env 循环 #1 → fast_io 循环 #1 → Popen swipe(1000ms) #1
t=0.15s:  game_env 循环 #1 → fast_io 循环 #2 → Popen swipe(1000ms) #2  (swipe #1 仍在执行!)
t=0.30s:  game_env 循环 #1 → fast_io 循环 #3 → Popen swipe(1000ms) #3  (swipe #1,2 仍在执行!)
t=0.45s:  game_env 循环 #2 → fast_io 循环 #1 → Popen swipe(1000ms) #4
t=0.60s:  game_env 循环 #2 → fast_io 循环 #2 → Popen swipe(1000ms) #5
t=0.75s:  game_env 循环 #2 → fast_io 循环 #3 → Popen swipe(1000ms) #6
t=0.90s:  game_env 循环 #3 → fast_io 循环 #1 → Popen swipe(1000ms) #7
t=1.05s:  game_env 循环 #3 → fast_io 循环 #2 → Popen swipe(1000ms) #8
t=1.20s:  game_env 循环 #3 → fast_io 循环 #3 → Popen swipe(1000ms) #9
t=1.35s:  所有循环结束
```

**结果**：9 个重叠的 `input swipe` 命令在约 1.5 秒内发出。由于 Popen 不等待，每个命令开始后立即返回，下一个命令马上发射。Android 设备收到一堆重叠的触摸事件，行为不确定——通常表现为快速连点而非持续按压。

### 2.3 修复

**fast_io.py** —— 改为单次同步调用：

```python
# 新版本
def long_press(self, x, y, duration_ms):
    subprocess.run(                      # run = 等待命令完成
        ["adb", "-s", self.serial, "shell", "input", "swipe",
         str(x), str(y), str(x), str(y), str(duration_ms)],
        capture_output=True,
        timeout=duration_ms / 1000 + 3,  # 1000ms + 3s buffer
    )
```

**game_env.py** —— 去掉外层循环：

```python
# 新版本
elif action_idx == 5:      # 长按
    ft.long_press(cx, cy, config.LONG_PRESS_DURATION_MS)
    # 不再循环，单次调用即可
```

**修复后时间线**：
```
t=0.00s:  adb shell input swipe cx cy cx cy 1000 开始
t=1.00s:  swipe 完成（ACTION_UP），subprocess.run 返回
t=1.00s:  Python 继续执行下一步
```

干净的一次按压 1 秒。没有重叠，没有连点。

### 2.4 技术原理

`adb shell input swipe x1 y1 x2 y2 duration_ms`：
- 在 (x1,y1) 发送 ACTION_DOWN
- 等待 duration_ms 毫秒（触摸保持）
- 在 (x2,y2) 发送 ACTION_UP

当 x1=x2, y1=y2 时，等价于"按住不动 duration_ms 毫秒"，即长按。

`subprocess.run` 阻塞等待 ADB 命令在设备上执行完毕后才返回。`subprocess.Popen` 只负责发射命令，不等待结果。
