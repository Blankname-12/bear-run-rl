# S4 开发流程全记录 —— 一步步怎么走过来的

## 起点：一个能跑但 PPO 不学的系统

**已有资产**：
- `label_game.py`：手动玩 + 键盘标注，10GB 数据，6 类动作
- `pretrain_cnn.py`：SimpleCNN 行为克隆，val_acc ~82%
- `play_ai.py --pretrain`：加载预训练模型，能躲 5+ 障碍物，存活 ~1 分 30 秒
- `train_ppo.py`：PPO 训练脚本，ActorCriticCNN，教师蒸馏奖励
- `fast_io.py`：minicap 截屏 + ADB input text 触控

**当时的问题**：PPO 训练 40 分钟，agent 只会输出 noop（不动）。

---

## 第一步：诊断"为什么只输出 noop"

### 观察到的现象

```
第 27 局 — 存活 35 步 | 奖励: -11.00
Step 18-37: 全部 动作=不动 奖励=+0.00（连续 20 步 noop）
Step 38: 红心丢失
Step 39: 死亡 -10.00
```

每局都这样：暖机 17 步 → 连续 noop 十几步 → 死 → 奖励 -11。

### 第一个猜测：奖励函数有问题

当时奖励：
```python
匹配教师 +1.0  /  不匹配教师 -0.5  /  死亡 -10.0
```

如果教师也预测 noop（因为标注数据 noop 多），agent 选 noop → 匹配教师 → +1.0。agent 选其他动作 → 大概率不匹配 → -0.5。

**验证**：查看教师预测日志：
```
Step 6:  右移 → 教师说"左移" -0.5
Step 7:  右移 → 教师说"左移" -0.5
Step 22: 滑铲 → 教师说"左移" -0.5
```

教师几乎总预测"左移"——但同一个模型在 `play_ai.py --pretrain` 中表现正常，能躲障碍物。

### 第二个猜测：教师预处理有问题

对比教师预处理 vs play_ai.py CNN 路径：
```python
# 教师 _teacher_predict
cropped = _crop_game_area(raw)
img = cv2.resize(cropped, (128, 128))
# → (3, 128, 128)

# play_ai.py CNN predict
cropped = _crop_game_area(raw)
img_small = cv2.resize(cropped, (128, 128))
# → (3, 128, 128)
```

一样！那为什么教师在 PPO 中表现这么差？

### 第三个猜测（关键发现）：教师和 agent 根本不在同一个输入空间

检查代码后才发现：

```python
# Agent (ActorCriticCNN) 的输入
# train_ppo.py RLGameEnv.get_state():
state = np.concatenate(self._frame_stack, axis=0)  # 4帧×3ch = 12ch, 320×320
lane = np.zeros((3, 320, 320))
state = np.concatenate([state, lane], axis=0)      # 12ch+3ch = 15ch, 320×320
# → ActorCriticCNN(input_channels=15)

# Teacher (SimpleCNN) 的输入
# _teacher_predict():
img = cv2.resize(cropped, (128, 128))
# → SimpleCNN(input_channels=3)
```

**15 通道 320×320 vs 3 通道 128×128 —— 完全不同的世界！**

再看权重加载：
```python
# load_pretrained_backbone 旧版逻辑
if ac_param.shape != pt_param.shape:
    if ac_name == "conv1.conv.weight":
        # (32, 15, 3, 3) vs (32, 3, 3, 3)
        ac_param[:, :3] = pt_param  # 只复制前 3/15 通道
        # 其余 12 个通道 → 随机初始化
```

**结论**：预训练权重基本没加载进去。Agent 约等于随机初始化 + 噪声教师奖励 → 坍缩 noop。

---

## 第二步：将 PPO 输入统一为 3ch×128×128

### 目标

让 Agent 和 Teacher 看到完全相同的"世界"。

### 改动

1. **config.py**：FRAME_STACK=4→1, FRAME_WIDTH/HEIGHT=320→128，删重复定义
2. **model.py**：ActorCriticCNN input_channels=15→3
3. **train_ppo.py RLGameEnv**：
   - 去 `_frame_stack` 列表 → 换 `_current_frame` 单帧
   - `get_state()` 从 `(15,320,320)` → `(3,128,128)`
   - 去跑道编码（lane channels）
4. **play_ai.py**：PPO 预测路径同样改为单帧 128×128
5. **load_pretrained_backbone**：简化为纯 1:1 映射（所有层形状匹配）

### 预期

- 卷积权重 8/8 层完整匹配
- Actor 头从 classifier 完整复制
- Agent 初始策略 ≈ 预训练模型行为

---

## 第三步：奖励函数第一次调整（noop=0）

改动后重新训练——还是坍缩！

但这次不是全 noop，而是**全滑铲**：
```
Step 17-22: 接连 6 步滑铲
Step 23:    偶尔匹配教师 +1.0
Step 24-38: 继续滑铲
```

### 分析

noop=0 后，noop 和所有其他动作都是 0 分（因教师仍预测不准）。策略随机收敛到第一个被采样的动作。

### 用户反馈

"谁教他一直滑铲的" → 教师没用，所有动作 0 分，和 noop 没区别。

---

## 第四步：去掉"不匹配惩罚"（-0.5→0）

```python
REWARD_TEACHER_MISMATCH = 0.0  # 原 -0.5
```

### 结果

仍然坍缩 noop。

日志：
```
Step 18-37: 全部 noop，奖励 0
Step 38: 红心丢失
Step 39: 死亡
```

### 分析

动作匹配教师 +1.0（但匹配不到），不匹配 0（和 noop 一样），动作可能导致死亡 -10.0。**不动的期望收益 > 乱动的期望收益**。

**用户精确总结**："做出动作就大概率扣分，然而不做动作不扣分，一会儿就出现保持不动了。"

---

## 第五步：去掉教师，纯环境奖励

既然教师不准，干脆不用。让环境本身提供信号。

```python
# 无教师
if action_idx == 4:    noop = 0
else:                  action = +0.1
DEATH = -10.0
```

### 结果

```
第 27 局: 奖励 -11.00, 存活 35 步
第 28 局: 奖励 -11.00, 存活 39 步
第 29 局: 奖励 -11.00, 存活 32 步
Avg20 = -10.57
```

几乎不变。agent 还是主要 noop，偶尔移动几步。

### 用户反馈

"这学个屁啊，根本就没学"

### 分析

所有非 noop 动作（跳/滑/左/右）都是 +0.1。正确跳跃和错误跳跃没区别。PPO 通过 GAE（γ=0.99, λ=0.95）把死亡 -10 回传，但 18 个游戏步骤（去掉暖机）中，信用分配纯粹是噪声——agent 无法知道到底是跳对了还是跳错了。

---

## 第六步：用户反馈"noop 不给分"已经做了

用户指出：noop=0 只解决了一半问题。教师虽然被去掉了，但没有替代的逐帧信号。PPO 依然学不会。

---

## 第七步：重新引入教师 + 优化奖励权重

### 关键认知变化

前面去掉教师是因为**教师不准**。但教师不准的根因是**输入格式不同**（已修复）。现在输入统一了，教师应该又准了。

### 新方案

```python
MATCH = +1.0     # 匹配教师 → 强正信号
MISMATCH = 0     # 不匹配 → 不罚
noop = -0.1      # 小惩罚防止坍缩
DEATH = -1.0     # 死亡轻罚（用户说 -10 "太重了，能不能改成扣 1 分"）
```

### 改进要点

1. Mismatch 0 不是 -0.5：探索不会被惩罚
2. noop 有 -0.1 负激励：避免"不动最安全"的局部最优
3. 死亡 -1.0 不是 -10.0：约 10 次匹配可抵一次死亡，不会扼杀探索
4. 教师回来了，但前提已满足（输入统一）

### 此时用户反馈

"红心丢失就不用扣分了，因为有滞后的，死亡扣分即可"

→ 去掉红心丢失扣分，只有死亡时扣 -1.0。

---

## 第八步：策略坍缩到定式循环

训练几局后出现新问题：agent 陷入"左左停右右停"循环，几乎从不滑铲。

### 用户描述

"现在几乎不存在下滑的动作了，并且模型的探索度不够啊，现在基本上出现了动作定式，左左停右右停循环往复"

### 分析

ENTROPY_COEF = 0.02 太小。PPO 的 entropy bonus 不足以对抗策略坍缩。Agent 找到了一个局部最优：
- 左移/右移视觉特征明显 → 预训练模型容易预测 → agent 容易匹配教师 → 得 +1.0
- 滑铲视觉特征不显著 → 预训练模型很少预测 → agent 即使尝试也拿不到奖励
- 左右交替 + 偶尔停 → 不会撞墙（边界有拦截）→ 稳定的正奖励流
- 策略越来越"尖锐"（softmax 集中在左/右/停 3 个动作上）

### 修复

```python
ENTROPY_COEF = 0.15  # 7.5x 提升
```

强制策略保持高熵，对抗坍缩。

## 第九步：数据增强（并行进行）

### 用户要求

"可以在预训练里尝试数据增强；我现在采集了 10 个 GB 的数据了"

### 第一次扩展

加了 7 种增强：水平翻转、随机裁剪、亮度、对比度、饱和度、高斯模糊、锐化。

### 用户反馈

"水平/垂直翻转都是没必要的，请你取消"

### 理由

Temple Run 类游戏画面永远不会镜像。水平翻转后的图（树反转、UI 反转）在真实游戏中不存在。在翻转图上学的特征反而误导模型。

### 最终方案

只保留 4 种：亮度抖动、对比度抖动、饱和度抖动、高斯模糊。全部是不改变空间结构的颜色/模糊增强。

### 同时加入

类别加权损失（样本少的滑铲/长按获得更高权重）。

## 第十步：play_ai 防抖策略

### 初始状态

play_ai.py 有激进的防抖：时间间隔 + 同动作阻断 + 置信度过滤。

### 用户第一步反馈

"能不能把这个防抖去掉？"

→ 我去掉了所有防抖，只保留长按 90% 阈值 + 低置信度过滤。

### 用户第二步反馈

"需要保留低置信度过滤，只需要删除防抖"

→ 恢复了低置信度过滤。

### 用户第三步反馈（最终要求）

"如果两次连续操作都是相同的，那就第二个舍弃，如果三个连续的，第三个及以后不舍弃"

→ 实现了**智能防抖**：
```
连续第 1 次 X → 放行
连续第 2 次 X → 跳过（防双击）
连续第 3 次 X → 放行（允许持续按住）
换动作       → 重置
```

## 第十一步：长按修复

### 用户问题

"为什么现在的长按 1 秒被识别成快速点击，根本就没有长按？？？？"

### 根因追踪

1. 读 `fast_io.py`：`long_press` 用 Popen + 循环 3 次
2. 读 `game_env.py`：`send_action(5)` 又循环 3 次
3. **3×3=9 个重叠的 `input swipe` 命令！**

Popen 不等待，命令还没执行完就发下一个。9 个重叠触摸事件被游戏当成快速连点。

### 用户咆哮

"是他妈跟你说发送 L 触发长按，就是按下，一秒后松开啊"

（我之前愚蠢地提议用 `input text L`，用户纠正说长按就是触摸按下→等→松开，不是键盘按键）

### 修复

- `fast_io.py`：单次 `subprocess.run`（同步等待，不是 Popen）
- `game_env.py`：去掉外层循环

## 核心教训

1. **迁移学习的格式对齐是不可妥协的前提**：输入差一个通道、一个像素，预训练权重就废了
2. **奖励设计的每一轮失败都有明确的因果逻辑**：不是"调参不 work"，是每一步坍缩都能解释为什么
3. **用户的直觉经常是对的**：死亡 -10 太重、翻转没必要、红心滞后无意义——这些反馈直接指向问题的物理原因
4. **教师蒸馏的前提是教师和学生同构**：不同输入空间的两个模型，教师信号就是噪声
5. **Entropy 是防止策略坍缩的最后防线**：太小就坍缩，太大就随机，需要反复调
